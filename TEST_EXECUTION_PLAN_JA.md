# CIE Platform — テスト実行計画

## 概要
本ドキュメントは、CIEプラットフォームがMVP要件を満たし、設計通りに動作することを検証するための6つの包括的なテストシナリオを定義しています。各シナリオは、`health_screening_japan.csv`テストデータセット（190レコード、150人のユニークな被験者 + 2年間の追跡データ40件）を使用します。

---

## テストデータセット概要
- **ファイル**: `test_data/health_screening_japan.csv`
- **レコード数**: 190（150件のベースライン + 40件の追跡調査）
- **被験者数**: 150人（患者ID: PID00001-PID00150）
- **調査年**: 2024年（ベースライン）、2025年（追跡調査対象）
- **変数**: 25項目（人口統計、生体指標、検査値）
- **個人識別情報**: 患者氏名（Layer 2 NLP検出テスト用）
- **データ品質**: 現実的な欠損値（2-5%）、外れ値、臨床的に妥当でない値を含む

**テスト用の主要変数**:
- 人口統計: 患者ID、患者氏名、施設コード、検査年、年齢、性別
- 生体指標: 身長_cm、体重_kg、BMI、収縮期血圧_mmHg、拡張期血圧_mmHg
- 血糖: 空腹時血糖_mg/dl、HbA1c_%
- 脂質: 総コレステロール_mg/dl、HDLコレステロール_mg/dl、LDLコレステロール_mg/dl、中性脂肪_mg/dl
- 肝機能: AST_U/L、ALT_U/L、γ-GTP_U/L
- 腎機能: クレアチニン_mg/dl、eGFR_ml/min、尿酸_mg/dl
- 尿検査: 尿蛋白、尿糖

## シナリオ1: 基本的なグループ比較（2群t検定）

**臨床的質問**: 
「男性と女性の被験者間で収縮期血圧に差があるか？」

**意図**:
```
データセット: health_screening_japan.csv
結果変数: 収縮期血圧_mmHg（数値型）
グループ分け変数: 性別（カテゴリ型: 男性、女性）
分析タイプ: 平均値の比較
期待される手法: 独立2標本t検定
```

**期待されるPlanner（計画立案者）の動作** (PL-001: 基本的な意図認識):
- 結果変数の特定: 収縮期血圧_mmHg
- グループ分け変数の特定: 性別
- 2グループの認識 → t検定（ANOVAではない）
- ペアリングの推定: なし（ベースラインのみ）

**期待される統計エージェントの動作**:
- 収縮期血圧_mmHgの欠損値検出（0であるべき）
- 正規性の仮定確認
- 適切なt検定の適用（パラメトリック or マン・ホイットニーU検定）
- グループ別の記述統計の生成

**期待される結果**:
- 記述統計: 性別ごとの平均収縮期血圧
- 検定統計量: t値、p値
- 差の信頼区間
- 効果量: Cohenのd

**検証基準**:
- ✓ 分析がエラーなく完了
- ✓ p値は両側
- ✓ 記述統計が手計算と一致
- ✓ 効果量（Cohenのd）が計算される
- ✓ 監査ログに記録: 意図、データプロフィール、手法選択

**実行時間予算**: 30秒

## シナリオ2: 相関分析（連続変数）

**臨床的質問**:
「BMIとHbA1cの間の相関は何か？」

**意図**:
```
データセット: health_screening_japan.csv
変数1: BMI（数値型）
変数2: HbA1c_%（数値型）
分析タイプ: 相関分析
期待される手法: 正規分布の場合ピアソン相関、歪んでいる場合スピアマン相関
```

**期待されるPlanner（計画立案者）の動作** (PL-001: 意図認識):
- 2つの数値変数を特定
- 相関分析要求の認識
- グループ分けなし → 単一相関

**期待される統計エージェントの動作**:
- HbA1c_%の欠損値検出（約5%予想）
- 欠損データの処理: リストワイズ削除またはインピュテーション
- 両変数の正規性確認
- ピアソン vs スピアマンの選択
- 相関係数とp値の計算
- 散布図の生成（可視化エージェント）

**期待される結果**:
- 相関係数（r または ρ）
- p値
- 相関の95%信頼区間
- 回帰直線付き散布図
- 残差診断

**検証基準**:
- ✓ HbA1c_%の欠損データを正しく処理
- ✓ ピアソンとスピアマン両方の結果を出力
- ✓ p値の解釈が正確
- ✓ 可視化が生成される
- ✓ 品質確認で相関が1.0に近い場合にフラグ（データ整合性チェック）

**実行時間予算**: 40秒

## シナリオ3: ペアード設計／反復測定（PL-004、PL-005にとって重要）

**臨床的質問**:
「追跡データを持つ被験者において、2024年から2025年にかけてHbA1cが有意に変化したか？」

**意図**:
```
データセット: health_screening_japan.csv
結果変数: HbA1c_%
グループ分け: 検査年（2024 vs 2025）
ペアリング: 患者ID（同一被験者の2回測定）
期待される手法: ペアードt検定
```

**期待されるPlanner（計画立案者）の動作** (PL-004: ペアード設計の検出):
- 反復する患者ID値の検出 → ペアード設計
- 被験者内構造の認識
- subject_id_varの特定 = 患者ID (PL-005)
- time_varの特定 = 検査年
- ペアード分析の選択

**期待される統計エージェントの動作**:
- ペアリングの検証: 各被験者が正確に2回の測定を持つ
- ワイド形式データの作成（ベースライン vs 追跡）
- 被験者内差の計算
- 差の正規性テスト
- ペアードt検定の適用（正規でない場合はウィルコクソン検定）
- 効果量の計算（ペアード用Cohenのd）

**期待される結果**:
- 記述統計: ベースライン、追跡、変化量
- ペアードt検定結果: t、自由度、p値
- 平均変化量 ± 標準偏差
- 変化量の95%信頼区間
- 効果量

**検証基準**:
- ✓ ペアード構造を正しく特定（PL-004検証）
- ✓ subject_id_var = 患者IDを正しく特定（PL-005検証）
- ✓ 1回の測定しかない被験者を除外
- ✓ ペアード分析のnを報告（150ではなく40であるべき）
- ✓ ペアード特有の統計（独立t検定ではない）
- ✓ 監査ログでペアリング検出を記録

**実行時間予算**: 35秒

## シナリオ4: 多変量分析（連続予測変数による重回帰）

**臨床的質問**:
「HbA1cを予測する要因は何か？ モデル: HbA1c ~ BMI + 年齢 + 空腹時血糖_mg/dl」

**意図**:
```
データセット: health_screening_japan.csv
結果変数: HbA1c_%（数値型）
予測変数: BMI、年齢、空腹時血糖_mg/dl（すべて数値型）
分析タイプ: 複数線形回帰
```

**期待されるPlanner（計画立案者）の動作** (PL-001: 複雑な意図):
- 複数の予測変数を解析
- 連続型の結果変数と予測変数を特定
- 複数線形回帰を選択
- グループ分けなし → 単一回帰モデル

**期待される統計エージェントの動作**:
- 多重共線性の検出（年齢/血糖のVIF確認）
- 予測変数の欠損値の処理
- 線形モデルの適合
- モデル診断の生成:
  - 係数 + 信頼区間 + p値
  - R² と調整済みR²
  - 残差プロット
  - 影響診断

**期待される補正の適用** (評価のシナリオ4より):
- 他の箇所で複数比較が検出された場合 → ボンフェローニ補正を適用
- 補正の根拠を記録

**期待される結果**:
- 回帰表: 各予測変数のβ、標準誤差、t、p値
- モデル概要: R²、調整済みR²、F検定
- 残差プロット: Q-Q、スケール-位置、残差 vs 適合値
- 新データに対する予測（要求された場合）

**検証基準**:
- ✓ 3つの予測変数を正しく特定
- ✓ VIF値が計算され確認されている（通常<5が許容）
- ✓ 血糖列の欠損値を処理
- ✓ R²の解釈が正確
- ✓ 残差診断が評価されている
- ✓ 監査ログでモデル式と仮定確認を記録

**実行時間予算**: 45秒

## シナリオ5: 品質ゲートのブロック（MVP にとって重要）

**臨床的質問**:
「データ品質に問題がある中でコレステロール差を分析する」

**設定**:
- health_screening_japan.csvをそのまま使用（外れ値を含む: インデックス5、10に極端なBP、インデックス10に極端なクレアチニン）
- 分析: 性別ごとの総コレステロール
- 予想されるデータ品質問題:
  - 臨床的に妥当でないクレアチニン（インデックス10で2.8 mg/dL）
  - 極端なHbA1c（インデックス5で15.2%）
  - 複数の列で欠損値

**期待されるデータ品質エージェントの動作**:
- HbA1c > 12%を重大な外れ値としてフラグ
- クレアチニン > 2.5を重大（重度の腎疾患）としてフラグ
- HbA1cの欠損（5%）を警告としてフラグ
- γ-GTPの欠損（5%）を警告としてフラグ
- eGFRの欠損（5%）を警告としてフラグ
- 尿蛋白の欠損（5%）を警告としてフラグ
- 品質レポート生成

**期待される品質確認画面**:
- フラグされたレコードと問題を表示
- 外れ値の可視化をレンダリング
- オプション提供:
  - 承認して進行（決定を記録）
  - 外れ値レコードを除外
  - 注意付きで進行

**期待されるレビュアーエージェント動作**（分析に進む場合）:
- レポートでフラグ: 「分析は臨床的に妥当でない値を含みます」
- 包含の根拠を記録
- 追加の臨床的レビューを推奨

**検証基準**:
- ✓ データ品質エージェントがすべての重大問題を特定
- ✓ 品質確認画面がナビゲート可能で、ユーザーが決定できる
- ✓ ユーザーが進行する場合: レビュアーが品質に関する注記を結果に追加
- ✓ ユーザーが外れ値を除外する場合: フィルタされたデータで分析実行
- ✓ 監査ログが品質決定を記録
- ✓ 結果が明確にデータ品質ステータスを記録

**実行時間予算**: 50秒（ユーザーインタラクション含む）

## シナリオ6: 個人識別情報（PII）検出と削除（セキュリティ重要）

**臨床的質問**:
「PII検出を有効にして分析する」

**設定**:
- health_screening_japan.csvを使用（現実的な日本人名を含む患者氏名）
- PlannerまたはDataQualityエージェントでPII検出を有効化
- データセット内で予想されるPII:
  - Layer 1（正規表現）: 患者IDパターン（PIDに続く5桁）
  - Layer 2（NLP）: 患者氏名（日本人名: 田中太郎、鈴木花子など）

**期待されるセキュリティエージェント/データ品質エージェント動作**:
- Layer 1検出: すべての患者ID値を特定（150ユニーク予想）
- Layer 2検出: 患者氏名列を人名として特定
  - NLPベースのセマンティック検出を使用
  - 日本人名辞書と比較
  - 信頼度スコア付きでフラグ
- PII報告生成:
  - 列: 患者氏名
  - リスク レベル: 高（個人識別可能な名前）
  - 影響レコード: 150（全部）
  - 推奨: 削除またはニックネーム使用

**期待される削除オプション**:
- 患者氏名をハッシュまたはニックネームに置き換え
- 患者氏名列全体を削除
- 保持するが監査ログでフラグ

**期待される結果**:
- 実際の名前を出力に送信せずに分析が進む
- 監査ログが記録: PII検出、削除方法、誰が、いつ
- レポートにPII注記が含まれる: 「個人識別可能な患者名を削除」

**検証基準**:
- ✓ Layer 1が患者IDを検出（すべての150レコード）
- ✓ Layer 2が患者氏名を検出（すべての150レコード、高信頼度）
- ✓ ユーザーが削除方法を選択可能
- ✓ 削除されたデータが分析で使用（元の名前ではない）
- ✓ 監査ログがPII処理を記録
- ✓ 出力レポートに実患者名が含まれない
- ✓ Capability Tokenライフサイクル: 発行 → 使用 → 失効（監査ログの証拠）

**実行時間予算**: 60秒

## 計測するパフォーマンス指標

各シナリオについて計測:

1. **Planner LLM応答時間**:
   - 意図認識とIntentObject生成の時間
   - トークン使用量（入力 + 出力）

2. **データ品質エージェント時間**:
   - データセット検証時間
   - 欠損値検出時間
   - 外れ値検出時間
   - PII検出時間（特にLayer 2 NLP）

3. **統計エージェント時間**:
   - 手法選択時間
   - Rコード実行時間
   - 計算時間

4. **可視化エージェント時間**:
   - プロット生成時間

5. **エンドツーエンドパイプライン時間**:
   - 意図入力から結果までの合計時間

6. **精度指標**:
   - 統計結果が手計算値と一致
   - 効果量が正確
   - p値が適切

7. **エラー率**:
   - エラーなく完了するシナリオ数
   - エラーの性質
