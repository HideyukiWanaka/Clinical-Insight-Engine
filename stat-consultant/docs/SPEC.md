# Stat Consultant — アプリ使用書（正典 / Single Source of Truth）

> このドキュメントは、実装中いつでも立ち返るための「完成アプリの全体像」です。
> 迷ったら常にこのSPECを優先します。個別の実装指示は `BUILD_PROMPTS.md` にあります。
> （フォルダ名 `stat-consultant/` は仮称。変更する場合はこの1行だけ直せば足ります）

---

## 1. これは何のアプリか

統計解析について、臨床研究者が**統計の専門家に気軽に相談するように**AIと対話し、
提案された**Rコードをワンクリックで自分のRStudioのスクリプトに挿入**できる、
個人利用のデスクトップ風チャットアプリ。

**設計思想**: 家電のように不要な機能は持たせない。Apple製品のように、3歳児でも
直感で使えるミニマルさを最優先する。前身プロジェクト（CIE Platform）の過剰な
作り込みが「エラー修正の無限ループ」を生んだ反省から、機能を徹底的に削ぎ落とす。

**コードの実行はしない**。実行は常にユーザー自身のRStudioで行う。本アプリは
「相談相手」と「コードの受け渡し役」に徹する。

---

## 2. ペルソナと3つの痛み

ターゲットは「かつての開発者自身」——Rコード1行を書くのにネット検索・書籍調査に
延々時間を溶かし、統計手法の選択に自信が持てず、思い通りの図を作れず結局Excelに
逃げていた、独学の臨床研究者。

このペルソナの**3つの痛み**にのみ向き合う（すべて同じ比重）:

| # | 痛み | 本アプリの応え |
|---|------|----------------|
| ① | **コード記述の壁** — 書き方を調べるのに時間を溶かす | 相談すると理由つきのRコードが返り、ワンクリックでRStudioに挿入 |
| ② | **手法選択への自信のなさ** — この検定で合っているか不安 | 実データの型・群数・欠損に基づいて前提条件を説明し、手法選択を支える |
| ③ | **ビジュアル化の妥協** — 思い通りの図が作れずExcelに逃げる | 論文等の図をアップロード→同テーマの図を実データで描くggplot2コードを生成 |

---

## 3. 差別化（ただのチャットAIとの違い）

会話品質そのものでは競わない。ハードに真似できない2本の柱で差別化する:

1. **実RStudio環境へのグラウンディング** — ユーザーが実際に読み込んでいるデータの
   列名・型・群数・欠損数をAIが把握した上で回答する。
2. **検証済み統計知識＋ユーザー参考資料へのグラウンディング** — 統計手法テンプレート
   （移植した知識）と、ユーザー自身がアップロードした参考資料（論文・ガイドライン・
   院内基準）に基づいて回答する。

## 4. 画面仕様

### 4.1 ウィンドウ形態
- ブラウザのアドレスバー・タブ・ブックマークバーを持たない**アプリモードウィンドウ**
  （Chrome `--app=` 起動、または PWA としてインストール）。
- OS標準の細いタイトルバー（閉じる/最小化/最大化）のみ残る。
- 中身は自前のReactフロントなので、配色・フォント・レイアウト・ダークモード・
  アイコン（PWAマニフェスト）は完全に自由。
- 将来、完全独自タイトルバーが必要になれば Tauri でデスクトップアプリ化（③段階）。

### 4.2 外観（確定案）
- **ベース**: Light & Clean（ミニマル）。オフホワイト背景、装飾なし、広い余白、
  フラットデザイン。視線が迷わない。
- **アクセントカラー**: Deep Teal 系を**1色だけ**、操作要素（送信ボタン・
  「RStudioへ送る」ボタン）にのみ使用。背景やコードカードは塗らない。
- **フォント**: システムUIサンセリフ。
- **ダークモード**: OSの設定に追従（デフォルトの見た目とは別軸、将来対応でよい）。
- **不採用**: Glassmorphic / Material You の多色グラデーション、Cute（キャラ装飾）系。
  臨床研究ツールとしての真面目さを損なうため。

### 4.3 UI要素（これだけ）
1. **チャット入力欄** — 通常のチャットとして何度でも往復できる。1メッセージで
   完結させる必要はない。
2. **添付ボタン1つ** — ファイル種別で用途を自動判別（ボタンを増やさない）:
   - 画像 → その場限りの「参考図」（そのターンのVision入力）
   - Markdown/テキスト → 永続化される「参考資料」（`POST /api/references`）
   - 添付直後にトーストで判別結果を表示（「参考資料として保存しました」/
     「今回の参考図として送信します」）。
3. **コードブロック毎の「RStudioへ送る」ボタン** — Addin接続時は挿入、未接続時は
   クリップボードへ自動コピー＋トースト。

### 4.4 メッセージ種別（最小セット）
`text`（ユーザー発言） / `image`（参考図プレビュー） /
`assistant_text`（一言理由つき、詳細は折りたたみ） / `assistant_code`。
※ intent/clarify/confirm/proposal/figures/manuscript 等の旧Planner由来種別は扱わない。

### 4.5 作らないもの（非スコープ）
会話履歴サイドバー、複数プロジェクト管理、モデル選択、設定画面、ログイン/認証UI、
ファイルツリー、コードエディタ／コンソールペイン、Dataset/Knowledgeモーダル。
（トースト通知のみ例外的に許容する軽量フィードバック）

## 5. 使用フロー

### 5.1 初回セットアップ（一度だけ）
1. バックエンドを起動（ダブルクリックで `uvicorn` を立ち上げるスクリプト）。
2. RStudioで一度 Addin を起動する。Addinはバックエンドが生成したローカルの
   共有シークレットファイルを自動で読むため、**トークンの手貼り付けは不要**。
3. アプリモードウィンドウでチャット画面を開く。

### 5.2 透明な環境同期（常時・自動）
Addinが起動中は、チャット送信の裏で自動的にRのグローバル環境をスキャンし
（`ls(.GlobalEnv)` と各オブジェクトの型・列名・型・欠損数・カテゴリ水準）、
バックエンドへ同期する。**ユーザー操作は一切不要**（手動更新ボタンは作らない）。
既存列から作った派生列（例: `df$bmi <- ...`）も、次回スキャンで通常列と同じく検出される。

### 5.3 痛み①のフロー（コード記述）
「〇〇群と△△群を比較したい」と相談 → 一言理由（例:「対応のないt検定／正規性を仮定」）
＋Rコードが返る → 「RStudioへ送る」で自分のスクリプトのカーソル位置に挿入。

### 5.4 痛み②のフロー（手法選択）
「この検定で合ってる？」と聞く → 同期済みの実データ（群が3つある・血圧列に欠損15件 等）
に基づいて前提条件を説明し、適切な手法を提案する。

### 5.5 痛み③のフロー（ビジュアル化）
論文の図を添付 → Vision が図のスタイルを解析 → 実データの列にマッピングした
ggplot2コードを生成 → 「RStudioへ送る」で挿入。

### 5.6 参考資料アップロード（随時）
統計の参考資料（Markdown/テキスト）を添付すると単一フォルダに保存され、以降の
回答のグラウンディングに使われる。承認フローや階層区別はない（個人利用のため）。

## 8. エンドポイント一覧

## 9. 環境同期ペイロード仕様とPII除外

### 9.1 ペイロード形
```json
{
  "objects": [
    {
      "name": "df",
      "class": "data.frame",
      "nrow": 120,
      "columns": [
        {"name": "血圧", "type": "numeric", "n_missing": 15},
        {"name": "group", "type": "factor", "n_missing": 0,
         "levels": [{"label": "Placebo", "count": 40},
                    {"label": "DrugA", "count": 41},
                    {"label": "DrugB", "count": 39}]}
      ]
    }
  ]
}
```
送るのは**集計値のみ**（列名・型・欠損数・カテゴリ水準ラベルと件数）。
個々の行データ（セルの値）は一切送らない（CLAUDE.md `inject_raw_data_rows=False` を遵守）。
欠損数・群数は個人を特定できない集計値であり、この禁止には該当しない。

### 9.2 PII除外
`PIIDetectorLayer1.detect(col_name, top_categories)`（`cie/security/pii_detector.py` から翻案）で
列名またはカテゴリ水準ラベルにPII所見が出た列は、型・欠損数・水準を**すべて**同期対象から
除外する。所見なしの通常の臨床変数名（「血圧」「年齢」等）はそのまま渡し、自然な会話を保つ。
**全列一律の var_n 匿名化はしない**（「血圧」等の自然言語参照が壊れるため）。
除外はAddin側（送信前）で行い、バックエンドでも二重チェックする。

## 10. やらないこと（スコープ境界の明示）

- サーバー側でのR実行（実行はユーザーのRStudioのみ）
- Orchestrator/Planner DAG、Capability Token の重い発行・失効機構
  （LLM呼び出しの try/finally ラップという最小形のみ移植）
- 3階層 Skill / Knowledge ガバナンス（METADATA.yaml・REGISTRY.yaml・人間承認・
  実行スナップショット不変性）
- 会話履歴サイドバー / モデル選択 / 設定画面 / 認証UI
- IDE型ワークベンチ（エディタ・コンソール・ファイルツリーの各ペイン）

## 12. 完了の定義（受け入れ確認）

3つの痛みに対応する一連の操作を通しで確認できること:
1. 「〇〇群と△△群を比較したい」→ コード＋理由が返り、RStudioに1クリックで挿入される。
2. 「この検定で合ってる？」→ 実データの型・群数・欠損に基づく前提説明が返る。
3. 論文の図を添付 → 実データ列にマッピングされたggplot2コードが生成される。
