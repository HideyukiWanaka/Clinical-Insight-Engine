# stat-consultant — 実機テストプラン

> このドキュメントは、サンドボックス環境では検証しきれなかった部分——実RStudio・
> 実LLM APIキー・実ブラウザを要する経路——を、開発者自身のマシンで確認するための
> 手順書。正典は `docs/SPEC.md`。個別の実装内容は `docs/BUILD_PROMPTS.md` を参照。

## 0. 概要

Step 0〜9 はすべて実装済みだが、Step 7（RStudio環境スキャン＋PII除外）・
Step 8（環境コンテキスト注入）・Step 9（参考図→ggplot2 Vision）は、直近の実装
セッションではサンドボックス環境（実RStudio無し・実LLM APIキー無し・実ブラウザ無し）
でしか検証できていない。具体的には以下がまだ本物の経路を一度も通っていない:

- 実RStudioへの Addin インストール（`remotes::install_local`）と Addins メニューからの起動
- `later` によるポーリングが実際のRStudioイベントループ上で動くこと
- 実データフレームを読み込んだ際の `ls(.GlobalEnv)` スキャンの実挙動
- 実LLM（Claude等）が環境コンテキスト・参考図を実際にどう解釈し応答するか
- ブラウザでの添付UI・トースト・コンポーザチップの実際の見え方と操作感

このテストプランは Step 7-9 を中心に、最後に SPEC §12 の3つの痛みの通し受け入れ
確認を行う。**対象プラットフォームは Mac/Linux**。Windows向けの手順・追加
チェックポイント（`rstudio_auth.py` にコメントされている「Pythonの
`Path.home()` (`USERPROFILE`) とRの `path.expand("~")` (`R_USER`/`HOME`) が
Windowsでは食い違いうる」というリスクの診断を含む）は
`docs/TEST_PLAN_WINDOWS.md` を参照。

LLMの応答は非決定的なので、検証基準は「特定の文言と完全一致」ではなく「意味的に
正しい内容（群数・欠損数への言及など）を含むか」で判定する。

## 1. 環境構築（前提）

### 1.1 backend

```bash
cd stat-consultant/backend
python3 -m venv .venv
.venv/bin/pip install -e .
.venv/bin/uvicorn app.main:app --reload --port 8000
```

確認: `curl localhost:8000/health` → `{"status":"ok"}`

起動ログに以下が出ることを確認（Step 6/7 の認証基盤）:
```
[stat-consultant] RStudio shared secret written to /Users/<you>/.stat-consultant/rstudio_token
```

### 1.2 frontend

```bash
cd stat-consultant/frontend
npm install
npm run dev            # backend を先に起動しておくこと
```

ブラウザで印字されたURLを開く。

### 1.3 APIキー

フロントの設定画面（右上の歯車アイコン）から Anthropic のAPIキーを登録する。
もしくは backend 起動前に `export ANTHROPIC_API_KEY=sk-...` でも可（フォールバック）。
モデルドロップダウンで Claude 系モデルが選択可能（灰色無効化されていない）ことを確認。

### 1.4 r-addin のインストール

```r
install.packages(c("rstudioapi", "httr2", "later"))  # 未インストールなら
remotes::install_local("stat-consultant/r-addin")     # または devtools::install(...)
```

> **注意（README.md との差分）**: リポジトリの `stat-consultant/README.md` は
> Step 6 時点の記述のままで、Addin名を「コード挿入: 開始／停止」と案内しているが、
> **Step 7 でリネーム済みの現在の正しい名前は「Stat Consultant: 開始」「Stat Consultant:
> 停止」**（`r-addin/inst/rstudio/addins.dcf` 参照）。本テストプランでは以後この
> 正しい名前で案内する。README.md 自体の追従修正はこのテストプランのスコープ外
> ——本プラン実施後に別途修正することを推奨する（§7 実行後チェックリストに記録欄あり）。

RStudioを再起動（または `.rs.restartR()`）した上で、Addinsメニューに
「Stat Consultant: 開始」「Stat Consultant: 停止」の2項目が出ることを確認する。

### 1.5 テスト用サンプルデータ

RStudioのコンソールで以下を実行し、PII列・通常列・高カーディナリティ列・欠損・
factor水準を一通り含むテスト用データフレームを用意する:

```r
set.seed(1)
df <- data.frame(
  患者名     = c("山田太郎","佐藤花子","田中一郎","伊藤次郎","阿部三郎",
                  "鈴木四郎","高橋五郎","渡辺六郎","中村七郎","小林八郎"),
  patient_id = sprintf("P%03d", 1:10),
  血圧       = c(120, NA, 130, 128, NA, 118, 135, NA, 122, 140),
  年齢       = c(45, 52, 60, 38, 71, 55, 49, 63, 58, 41),
  group      = factor(rep(c("Placebo","DrugA","DrugB"), length.out = 10)),
  memo       = c("特になし","経過良好","要観察","","",
                  "aaa","bbb","ccc","ddd","eee")
)
```

- `患者名` / `patient_id` → PIIとして完全除外されるべき列。
- `血圧` / `年齢` / `group` → 通常どおり同期されるべき列（`group` は3水準）。
- `memo` → 列名の「メモ」パターンに一致し除外されるべき列（実際に自由記述が
  混じっていても除外されるが、今回は列名だけでも除外条件を満たす）。

## 2. Step 7 シナリオ群（環境スキャン＋PII除外）

各シナリオの前提: 1章の環境構築が完了し、backend/frontend が起動している。
Addin未起動の場合は先に RStudio の Addins メニューから **「Stat Consultant: 開始」**
を実行する（起動メッセージ「Stat Consultant: コード挿入と環境同期を開始しました」が
コンソールに出る）。

### S7-1: 基本同期の到達確認

- **手順**: 1.5 の `df` をRStudioのGlobalEnvに読み込んだ状態で、2〜3秒（ポーリング
  既定間隔2秒×数サイクル）待つ。
- **期待される結果**: backendの起動コンソールに
  `[stat-consultant] environment sync: 1 object(s) [df(data.frame, 10 rows, N cols)]; ...`
  という行が出力される。
- **検証基準**:
  - ✓ ログが実際に出力される（＝Addin→backendの通信が確立している）
  - ✓ `N` が「PII列を除いた列数」になっている（1.5のdfは6列だが、`患者名`・
    `patient_id`・`memo` の3列が除外されるはずなので `N=3` 前後になる）

### S7-2: PII列除外の確認

- **手順**: S7-1 の状態で、backendのログに出た列数、および Step 8 のグラウンディング
  （S8-1参照）でチャットに聞いた際の応答内容を確認する。
- **期待される結果**: `患者名`・`patient_id`・`memo` のいずれについても、backend側の
  グラウンディング情報（Step 8 で注入されるプロンプト内容）に一切現れない。
- **検証基準**:
  - ✓ チャットで「私のデータにどんな列がありますか？」のように尋ねた際、応答に
    `患者名`・`patient_id`・`memo` が一切言及されない
  - ✓ 一方で `血圧`・`年齢`・`group` は言及される（除外し過ぎていないことの確認）

### S7-3: 通常列・factor水準の正確性

- **手順**: S7-2 のチャット応答、または（余裕があれば）backendを一時的に
  `--log-level debug` 等にせず、単純に `curl` で `/api/environment/sync` の受信直後の
  ログ行の列数のみで確認してもよい。より厳密に確認したい場合は、backend側に
  一時的な `print(request.app.state.environment.latest)` を **テストのためだけに**
  仮追加してコンソールで中身を見る、という手も可（本番コードには残さない）。
- **期待される結果**: `group` 列が `type: "factor"` で、`levels` に
  `Placebo` / `DrugA` / `DrugB` がそれぞれ約3〜4件ずつの件数付きで入っている。
  `血圧` は `n_missing: 3`（元データのNA数と一致）。
- **検証基準**:
  - ✓ 水準ラベルと件数が実際のデータと一致する
  - ✓ `n_missing` が実際のNA数と一致する

### S7-4: 派生列の検出

- **手順**: S7-1〜S7-3 確認後、RStudioコンソールで
  `df$bmi <- c(22,24,26,21,28,23,25,27,22,24)` を実行し、次のポーリングサイクル
  （数秒）を待つ。
- **期待される結果**: backendログに新しい同期行が出力され、`bmi` 列が加わる
  （列数が1つ増える）。チャットで確認した場合、`bmi` にも言及できるようになる。
- **検証基準**:
  - ✓ コード実行から数秒以内に新しい同期ログが出る
  - ✓ 追加した `bmi` 列が同期対象に含まれる（PII除外パターンに掛からない通常列
    なので除外されないはず）

### S7-5: 変化なしサイクルでの再送抑制

- **手順**: S7-4 の後、何もデータを変更せずに30秒〜1分程度放置する。
- **期待される結果**: backendログに同期行が連続して出続けない（環境シグネチャが
  変化していないサイクルはPOSTされない設計）。
- **検証基準**:
  - ✓ 変更なしの間、同期ログが（再)出力されない
  - ✓ 何かデータを変更した直後には再度ログが出る（S7-4の逆確認）

### S7-6: ソースドキュメント未オープンでの同期継続

- **手順**: RStudioで開いているすべてのソースファイルタブを閉じる（コンソールのみの
  状態にする）。その状態で `df$note <- "x"` のような軽微な変更を加える。
- **期待される結果**: コード挿入機能（Step 6, `poll_once`）は「アクティブな
  ソースドキュメントが無い」ため何もしないが、環境同期（`env_scan_once`）は
  ソースドキュメントの有無に関係なく動作し、変更が同期される。
- **検証基準**:
  - ✓ ソース未オープン状態でも新しい同期ログが出る

### S7-7: backend再起動時の自己回復

- **手順**: backendプロセスを一度 Ctrl+C で止め、再度
  `uvicorn app.main:app --reload --port 8000` で起動し直す（新しいトークンが
  `~/.stat-consultant/rstudio_token` に書き込まれる）。Addin側は**再起動しない**。
  その状態でRStudio側のデータを少し変更する。
- **期待される結果**: 最初の1〜2サイクルは401エラーが出うるが、Addin側は次の
  ポーリングでトークンファイルを再読込するため、Addinを手動で再起動しなくても
  数秒以内に同期が復帰する。
- **検証基準**:
  - ✓ Addinを再起動せずに同期が自動復帰する
  - ✓ RStudio側のコンソール（`message()`出力）にエラーが出ても、ポーリング自体は
    止まらず継続する

## 3. Step 8 シナリオ（環境コンテキスト注入）

### S8-1: 実データに基づく手法提案（BUILD_PROMPTS Step 8 の完了確認そのもの）

- **前提**: 2章のシナリオで `df`（3水準の `group`、欠損混じりの `血圧`）が同期済み。
- **手順**: チャットで「群間比較したいんだけど、この検定で合ってる？」のように
  具体的な列名を出さずに質問する。
- **期待される結果**: 応答が実際の `group` の水準数（3群）に言及し、一元配置分散
  分析（ANOVA）系の手法を候補に挙げる。`血圧` の欠損数（3件）にも触れ、除外か
  多重代入かの助言がある。
- **検証基準**:
  - ✓ 応答が「3群」「3水準」等、実際の群数に言及している
  - ✓ 欠損数（3件、または「欠損がある」という言及＋対処の助言）に触れている
  - ✓ 生成されたRコードがあれば、実際の列名（`血圧`、`group`）をそのまま使っている
    （プレースホルダのままではない）

### S8-2: 未同期状態でのハルシネーション防止

- **手順**: 新しい会話（別のconversation、またはbackend再起動で環境未同期の状態）で、
  データを一切読み込まずに「私のデータの列で相関を見たい」のように尋ねる。
- **期待される結果**: 応答が実在しない列名を捏造せず、一般的な進め方の説明や
  「どんな列がありますか？」といった聞き返しにとどまる。
- **検証基準**:
  - ✓ 存在しない具体的な列名（例: 適当な `x1`, `score` 等）を実データであるかの
    ように断定的に使わない

## 4. Step 9 シナリオ（参考図→ggplot2 / Vision）

### S9-1: 添付UIの見た目・トースト

- **手順**: 論文の図やウェブ上の箱ひげ図画像を1枚用意し、チャット下部の
  添付ボタン（クリップ）から選択する。
- **期待される結果**: トースト「今回の参考図として送信します」が出て、コンポーザ
  上部にサムネイルチップ（画像プレビュー＋「今回の参考図」ラベル＋×で取り消し）が
  表示される。
- **検証基準**:
  - ✓ トーストが正しいタイミング・文言で出る
  - ✓ チップの×ボタンで添付を取り消せる（取り消し後、送信ボタンはテキストが
    空なら再び無効化される）

### S9-2: 実データへのマッピング生成

- **前提**: 2章のデータが同期済み。
- **手順**: 群別の箱ひげ図画像を添付し、テキストなしで（または「こんな感じの図が
  欲しい」を添えて）送信する。
- **期待される結果**: ユーザー吹き出しに添付画像のプレビューが表示され、応答として
  実際に同期されている列（`group`、`血圧` 等）にマッピングされた ggplot2 コードが
  code ブロックで返る。理由（reason）にどの列をx/y/群に対応させたか一言ある。
- **検証基準**:
  - ✓ 生成コードが `ggplot(df, aes(x = group, y = 血圧, ...))` のように実列名を使う
  - ✓ 図に無い具体的な数値を捏造していない（コード内で実データを参照する形に
    なっている。ハードコードされた数値の箱ひげ統計量などが無い）

### S9-3: 画像の非永続化確認

- **手順**: S9-2 の送信後、ブラウザをリロードする。
- **期待される結果**: 会話履歴は復元されるが、送信した画像そのものが
  サーバー側に保存された形跡がない（`stat-consultant/backend` 配下に画像ファイルが
  作られていない。`ls stat-consultant/backend/user_references/` に画像が無いことを
  確認——このフォルダは参考資料＝Step4用で、画像は最初から対象外のはず）。
- **検証基準**:
  - ✓ `user_references/` フォルダに画像ファイルが存在しない
  - ✓ （余裕があれば）ブラウザのlocalStorageに保存されているのは会話の
    「表示用」画像URL（dataURL、クライアント側キャッシュ）のみで、サーバーには
    残っていないことを理解した上で許容する（SPEC上「保存しない」の意味は
    サーバー側非永続化であり、フロントの表示用ローカルキャッシュは対象外）

### S9-4: 生成コードのRStudio挿入（Step 6との結合）

- **手順**: S9-2 で返ったコードカードの「RStudioへ送る」ボタンを押す。
- **期待される結果**: Addinが起動していれば、開いているRStudioスクリプトの
  カーソル位置に ggplot2 コードが挿入される。未接続ならクリップボードにコピーされ
  トーストが出る（Step 5のフォールバック）。
- **検証基準**:
  - ✓ Addin起動中は数秒以内にカーソル位置へ挿入される
  - ✓ 挿入されたコードを実行して実際にプロットが描画できる

## 5. SPEC §12 受け入れ確認（3つの痛みの通し確認）

Step 7-9 個別確認がすべて完了したら、1つの連続した会話として以下を通しで実施する
（BUILD_PROMPTS.md 付録「全体の検証」に対応）。会話をリセットせず、同じ
conversation内で①→②→③の順に行う。

1. **① コード記述の壁**: 「group の Placebo と DrugA で血圧を比較したい」と相談する。
   → 一言理由（例:「対応のないt検定／正規性を仮定」）＋Rコードが返り、
   「RStudioへ送る」で自分のスクリプトのカーソル位置に1クリックで挿入される。
2. **② 手法選択への自信のなさ**: 「この検定で合ってる？」と聞く。
   → 実データの型・群数（3群であること）・欠損（`血圧` に3件）に基づく前提説明が
   返り、①の二群比較ではなく本来ANOVA相当を検討すべき旨（3群あるため）に触れるなど、
   実データを踏まえた具体的な指摘がある。
3. **③ ビジュアル化の妥協**: 論文の図を添付する。
   → 実データの列にマッピングされたggplot2コードが生成され、「RStudioへ送る」で
   挿入できる。

- **検証基準**:
  - ✓ 3つのシナリオが同一会話内でコンテキストを保持したまま完結する
    （②で①の内容を踏まえた発言があるなど、会話の連続性がある）
  - ✓ 3つとも「RStudioへ送る」→実際のRStudioへの反映まで確認できる
