# TEST_PLAN.md 実施中に見つかった問題点

`docs/TEST_PLAN.md` §6「実行後チェックリスト」の「見つかった問題点の一覧化（再現手順つき）」に対応する記録。実施中に見つかった問題をここに追記し、後でまとめて修正判断する。

## 未対応

- **軽微・対応保留**: 群ごとの有効データ数（非欠損数）が極端に少ない／ゼロの場合の但し書きルール（`fewshot.py`2群比較セクション、既存）が、同じセクション内の「群数構造への言及」ルールと競合し、毎回両方は拾われない（どちらか一方のみ言及されることがある）。プロンプトのさらなる調整より、「LLMの提案は仮説、RStudioでの実行結果が最終確認」という設計上の安全網に委ねる方針とし、これ以上のプロンプト調整はしない（ユーザーと合意済み）。

## 対応済み

- **連続する「RStudioへ送る」挿入で改行が無くなる**（複数のコードブロックを別々のタイミングで挿入すると、直前の挿入の末尾に次のコードが改行無しで連結される）→ 原因は`rstudioapi::insertText()`がカーソル位置に文字どおり挿入するだけで、既にドキュメントに何が挿入済みかを考慮しない設計だったため。同一ポーリングサイクル内で複数ブロックをまとめて挿入する場合は`"\n\n"`区切りだったが、別々のポーリングサイクルをまたぐ挿入には区切りが無かった（Step 6導入時からの既存の設計上の抜け、リグレッションではない）。`stat-consultant/r-addin/R/insert.R`の`poll_once()`で、挿入テキストの先頭に常に`"\n\n"`を付与するよう修正。**実機で検証済み**: 複数コードブロックを連続して「RStudioへ送る」→構文エラー無く連続実行でき、エディタ上でも空行区切りになっていることを確認。
- SPEC §12①：droplevels修正後の最終動作確認（DrugA群は血圧が全欠損のため、代わりにPlacebo vs DrugBで再テスト）→ **実機で検証済み**: droplevelsで正しく2群に絞り込み、Shapiro-Wilk・Welchのt検定・Cohen's dまで一連の流れがエラー無く完走した。
- SPEC §12②：手法確認時に、同期データの群数構造への言及が欠落（`group`が実際には3水準あるのに、2群比較の範囲を確認せずコードを出してしまう）→ 最初`prompts.py`の`_OUTPUT_FORMAT`に一般的なルールとして追加したが3回連続で効果が出ず、`fewshot.py`の「2群比較」セクション（droplevelsルールが既に効いていた場所）に具体例つきの指示として移設・強化して解決。**実機で検証済み**: 「groupには3水準あるが今回はPlaceboとDrugAの2群で良いか」と自発的に確認する応答を確認。
  - **切り分けで見つかった副次的な症状（対応不要、記録のみ）**: 検証の過程で一度、環境が同期されているはずなのにチャットが実列名の代わりに汎用プレースホルダ（`df_original`, `blood_pressure`）を使い、「群の数は直接見えない」とハルシネーションする現象が発生した。原因はGlobalEnvの同期対象オブジェクトが`df`本体の内容を変えないまま増減しても、Addinのポーリングが停止していたりシグネチャが変化しなければ再送されない、という既知の同期タイミングの問題（本セッション序盤で修正した`.state$last_env_sig`の仕組みと同根）。Addinsメニューから「Stat Consultant: 開始」を再実行し強制フル再同期させることで解消した。コードのバグではなく運用上の注意点（GlobalEnvを大きく変更したら明示的に「開始」を再実行して同期を確実にする）として記録。

- droplevels漏れによる `grouping factor must have exactly 2 levels` エラー（SPEC §12①）→ `fewshot.py`に修正済みルールを追加（本セッションで対応）。
- SPEC §12③：参考図から読み取ったラベル（`Sleep`/`ST`/`LPA`/`MVPA`等）を、同期済みデータに存在しないにもかかわらず実列名として使ってしまう（架空列名の創作）→ `prompts.py`の`IMAGE_INSTRUCTION`に、コード生成前に同期済み列名一覧と照合する手順、代用時の明示義務、該当列が無ければコードを出さず質問で返す義務を明記して修正済み。修正後、実機で「該当する列が無いので教えてほしい」と正しく質問で返す挙動を確認（本セッションで対応）。
- 日本語列名を含むggplot2プロットの文字化け（S9-4）→ 1件ずつの都度パッチ（macOS固有の`base_family = "HiraginoSans-W3"`ハードコード）は環境依存で恒久対応にならないため見送り、代わりに`fewshot.py`に新設した「可視化（ggplot2）でよくある落とし穴」セクションで、クロスプラットフォームな`showtext`パッケージ（CRAN公式）の使用を案内するルールを追加。**実機で検証済み**: `library(showtext); showtext_auto()`を使ったコードが生成され、日本語ラベルが正しく描画されることを新規会話で確認。
- SPEC §12③（応用）：積み上げ棒グラフで個別行が未集計のまま積み上がる（`pivot_longer`後の生データをそのまま`geom_bar(position="fill")`に渡し、1患者=1スライスになる）→ 同じく`fewshot.py`の新セクションに、ggplot2公式ドキュメント（geom_bar/geom_colのstat_count/stat_identityの違い）とR Graph Galleryの標準パターンに基づき、「`group_by()` + `summarise()`で集計してから`geom_col(position="fill")`に渡す」ルールを追加。`prompts.py`の`IMAGE_INSTRUCTION`からもこのルールを参照するようクロスリンクした。**実機で検証済み**: `group_by() + summarise()`による集計を経てから`geom_col(position="fill")`に渡すコードが生成されることを確認。
  - **注記（新たな軽微な発見）**: 修正直後、同じ会話内で再テストしたところ集計ルールは効いたがフォントルールは効かず、依然として`"MS Gothic"`等のOS固有フォント名をハードコードした。原因は、その会話内に以前（S9-4診断時）私が直接指示した`base_family = "HiraginoSans-W3"`という過去のやり取りが会話履歴として残っており、新しいシステムプロンプトのルールより優先されたためと考えられる。**新しい会話（会話リセット）で再テストしたところ両ルールとも正しく機能した**。これはコードのバグではなくLLMの一般的な性質（同一会話内の直近の会話履歴がシステムプロンプトの一般ルールより優先されやすい）によるものなので、対応不要（記録のみ）。
