## 1. UI設計思想の概要

CIE Platformのプレゼンテーション層は、「統計の専門家でなくても、臨床研究の文脈を理解していれば迷わず使える」という一点を最優先に設計する。UIはAIの能力を誇示する場ではなく、研究者の思考プロセスを支援する静かなインターフェースである。RStudioのペイン構成、Google Docsの共同編集体験、REDCapのフォーム設計が持つ「臨床研究者にとっての既視感」を意図的に取り入れ、学習コストを最小化する。同時に、患者データを扱うシステムとして、PII操作・人間承認・セキュリティ状態を視覚的に明確に伝える義務を負う。AIはあくまで提案し、人間が決定する——この関係性はUIのすべての層に一貫して表現される。

## 2. 全体レイアウト構成

CIEは**3ペイン構成**を基本とする。RStudio・VS Codeで培われた研究者の空間認識を活用し、「何がどこにあるか」の学習コストをゼロに近づける。

```
┌─────────────────────────────────────────────────────────────────┐
│  [ステータスバー]  プロジェクト名 │ 実行ID │ 接続状態 │ セキュリティ │
├──────────────┬──────────────────────────────┬────────────────────┤
│              │                              │                    │
│  左ペイン    │       中央ペイン             │   右ペイン         │
│  (240px)     │       (fluid)                │   (320px)          │
│              │                              │                    │
│ ▸ プロジェクト│  [メインコンテンツエリア]    │ [コンテキストパネル]│
│   ナビゲーション│                            │                    │
│              │  ● 入力フォーム              │ ● 実行ステータス   │
│ ▸ ワークフロー│  ● ワークフロービジュアライズ│ ● 人間承認リクエスト│
│   ステップ   │  ● 結果プレビュー            │ ● 監査ログ         │
│              │  ● レポートエディタ          │ ● データ品質サマリ │
│ ▸ ファイル   │                              │ ● Agentアクティビティ│
│   ブラウザ   │                              │                    │
│              │                              │                    │
└──────────────┴──────────────────────────────┴────────────────────┘
│  [フッター]  オフライン状態 │ Rバージョン │ 最終保存 │ ショートカット │
└─────────────────────────────────────────────────────────────────┘
```

### ペインの役割分担

| ペイン | 役割 | 参照UI |
|--------|------|--------|
| 左ペイン | プロジェクト構造・ワークフロー進捗・ファイル管理 | VS Code Explorer、RStudio Files |
| 中央ペイン | 主要作業エリア。入力→実行→結果→レポートの流れに沿って画面が切り替わる | Jupyter Notebook、Google Docs |
| 右ペイン | コンテキスト情報。実行状態・承認待ち・ログ・品質情報を常時表示 | RStudio Environment/Console |
| ステータスバー | システム全体の状態を一行で表示。異常時は色でアラート | VS Code ステータスバー |

### ペイン折りたたみ規則

- 左ペインは `Ctrl+B` で折りたたみ可能（作業集中モード）
- 右ペインは承認待ちタスクが存在する場合は自動展開・折りたたみ不可
- モバイル表示では3ペインをタブ切替に変換する

## 3. 主要画面一覧と役割

### SCR-01: プロジェクトダッシュボード

**役割:** CIEを開いた最初の画面。進行中のプロジェクト一覧と最終作業状態を表示する。新規プロジェクト作成のエントリポイント。

**設計方針:**
- REDCapのプロジェクト一覧に近い構造
- 各プロジェクトカードに「最終ワークフロー状態」「承認待ちタスク数」「品質スコア」を表示
- 完了・進行中・失敗の3状態をアイコンと色で区別

**主要コンポーネント:** ProjectCard, WorkflowStatusBadge, ApprovalCounter

---

### SCR-02: 研究意図入力画面（Intent Entry）

**役割:** ユーザーが自然言語で研究目的を記述し、Planner Agentがintent_objectを生成するメインエントリポイント。AP-001 Intent Firstの中核。

**設計方針:**
- 大きなテキストエリアを画面中央に配置（Googleドキュメントの新規作成に近い空白感）
- プレースホルダーで入力ガイドを提示（例:「治療群Aと対照群Bの術後90日死亡率を比較したい」）
- データセットアップロードエリアをテキストエリアの下に配置
- Planner Agentの解釈結果（intent_object）をリアルタイムで右ペインに表示
- 解釈に自信がない項目（confidence_score < 0.7）は黄色でハイライト

**人間承認ポイント:** intent_objectの確認・修正ステップを必須とする。「この解釈で進める」ボタンを明示的にクリックするまでワークフローは進まない。

**主要コンポーネント:** IntentTextarea, DatasetDropzone, IntentPreviewPanel, ConfidenceIndicator

---

### SCR-03: ワークフロービジュアライザー（Workflow View）

**役割:** workflow.yamlに定義されたDAGをインタラクティブなステップ表示として可視化する。「今どこにいるか」「次は何が起きるか」を一目で把握できる。

**設計方針:**
- 水平方向のステップチェーン（Jupyter Notebookのセル構造に近い）
- 各ステップを「未実行 / 実行中 / 完了 / 失敗 / 承認待ち」の5状態で表示
- 現在実行中のAgentをアニメーションで示す（スピナー）
- 承認待ちステップは右ペインの承認UIと連動してハイライト
- クリックで各ステップのIOペイロードをモーダル表示（AP-018 Observable Systems）

**主要コンポーネント:** WorkflowDAGView, StepStatusIndicator, AgentActivityFeed, StepDetailModal

---

### SCR-04: データ品質レポート画面（Data Quality Review）

**役割:** Data Quality Agentが生成したquality_reportをユーザーに提示し、Critical Issueの解消または明示的な承認を得る人間承認ポイント。

**設計方針:**
- 問題の深刻度（Critical / Warning）をセクション分けして表示
- 各変数の欠損率をバーチャートで視覚化（20%閾値ラインを赤線で表示）
- Critical Issueが残る場合は「次へ進む」ボタンを非活性化
- var_nエイリアスとオリジナル列名の対応を表示する専用パネルを右ペインに配置
  （ただしPII保護のため、オリジナル列名は「* * *」マスク表示 → Security Agent承認後に展開）

**人間承認ポイント:** 全Critical Issueの解消または明示的な「理解した上で進む」チェックボックス入力が必須。

**主要コンポーネント:** QualityGatePanel, MissingValueChart, CriticalIssueList, ColumnAliasPanel

---

### SCR-05: 統計解析設定・実行画面（Analysis Configuration）

**役割:** Planner/Statistics Agentが生成したanalysis_planを研究者に提示し、方法論の妥当性を確認した上で実行承認を得る。

**設計方針:**
- 選択された統計手法を平易な日本語で説明（例:「正規性が確認されたため、Welch検定を使用します」）
- 仮定チェック結果をチェックリスト形式で表示（REDCapのバリデーションリストに近い）
- 選択手法の変更UIを提供するが、変更理由の記入を必須とする（監査証跡）
- security_reviewノードに到達したら「実行を承認する」ボタンを右ペインに大きく表示
- ボタンクリック前に確認ダイアログ（「この操作はRスクリプトを実行します」）

**人間承認ポイント:** security_review承認。ボタンは意図的に1クリックでは押せない二段階確認。

**主要コンポーネント:** AnalysisPlanCard, AssumptionChecklist, MethodSelector, ExecutionApprovalPanel

---

### SCR-06: 結果・レポート編集画面（Results & Report）

**役割:** 解析実行後の結果表示、図表プレビュー、および原稿草稿の確認・編集。最終的な人間承認とエクスポートのエントリポイント。

**設計方針:**
- 数値結果・図・原稿テキストを並べて表示（Google Docsのコメント機能に近い構成）
- 自動生成テキストにはAIアイコン（🤖）を付与し、人間が書いた部分と明確に区別
- unresolved_itemsは黄色のコメントアノテーションとして本文中に表示
- 数値のトレーサビリティ機能: 原稿中の数値をクリックすると、対応するexecution_resultの値を右ペインで参照できる
- エクスポートボタンは右ペインの承認パネルに配置し、reviewerスコア≥90の場合のみ活性化

**人間承認ポイント:** レポートエクスポート承認（permissions.yaml: `report.export_external`）。

**主要コンポーネント:** ResultsPanel, FigureGallery, ManuscriptEditor, TraceabilityPopover, ExportApprovalPanel

---

### SCR-07: 監査ログ・観測画面（Audit & Observability）

**役割:** 実行済みワークフローの完全な監査証跡を提供する。AP-018 Observable Systemsの視覚的実装。

**設計方針:**
- タイムライン形式でワークフロー全実行履歴を表示
- 各エントリをクリックすると入出力ペイロードを展開表示
- セキュリティイベント（WARNING / CRITICAL / BREACH）は色分けで強調
- フィルタリング：Agentごと・状態ごと・時間範囲で絞り込み可能
- エクスポート機能（監査用CSVダウンロード）

**主要コンポーネント:** AuditTimeline, EventDetailPanel, SecurityEventBadge, LogFilterPanel

## 4. デザイン原則

### カラーパレット

CIEは「信頼性・精密さ・落ち着き」を伝える配色を採用する。派手さより読みやすさを優先する。

```
Primary:
  --cie-blue-700:    #1D4E89   // ヘッダー・プライマリボタン・リンク
  --cie-blue-500:    #2E74C0   // ホバー状態
  --cie-blue-100:    #DBEAFE   // ハイライト背景・選択状態

Neutral:
  --cie-gray-900:    #111827   // 本文テキスト
  --cie-gray-600:    #4B5563   // サブテキスト・ラベル
  --cie-gray-200:    #E5E7EB   // ボーダー・区切り線
  --cie-gray-50:     #F9FAFB   // パネル背景

Status:
  --cie-success:     #059669   // 完了・合格
  --cie-warning:     #D97706   // 警告・要確認（quality warningなど）
  --cie-critical:    #DC2626   // Critical Issue・セキュリティ違反
  --cie-approval:    #7C3AED   // 承認待ちアイテム（紫: 人間アクション必要）

AI-indicator:
  --cie-ai-teal:     #0D9488   // AI生成コンテンツのアクセント（落ち着いたティール）
```

**承認待ち状態に紫（`--cie-approval`）を使用する理由:** 赤（エラー）・黄（警告）と意味が重複しない固有色を割り当てることで、「これは人間が行動する必要がある」という状態を他の全状態から即座に区別できる。

### フォント

```
見出し・UI要素: "Inter", -apple-system, BlinkMacSystemFont, sans-serif
  サイズ体系: 12px / 14px / 16px / 20px / 24px / 32px
  ウェイト体系: Regular(400) / Medium(500) / SemiBold(600)

コード・数値・ログ: "JetBrains Mono", "Cascadia Code", monospace
  使用箇所: execution_result数値・Rスクリプト表示・監査ログ・var_nエイリアス

和文フォント: "Noto Sans JP", "Hiragino Kaku Gothic ProN", sans-serif
  使用箇所: 日本語UIラベル・説明文・エラーメッセージ
```

### トーン

- **UIラベル:** 簡潔な体言止め（「データ品質チェック」「統計手法の選択」）
- **説明テキスト:** 平易な日本語。統計用語には初出時に括弧で英語を付記（例:「信頼区間（95% CI）」）
- **エラーメッセージ:** 何が起きたか + 何をすればよいか の2文構成。技術的エラーコードは折りたたみで提供
- **AIの発言:** 一人称を使わない。「〜が検出されました」「〜を選択しました（理由: ）」の形式
- **承認ダイアログ:** 「〜してもよいですか？」ではなく「〜を実行します。確認してください。」の能動表現

## 5. セキュリティ・Human Authorityの表現方法

### 原則: セキュリティ状態は「見えない」ことが最大のリスク

CIEは患者データを扱う研究支援ツールである。セキュリティと人間承認の状態は、常に視覚的に明確でなければならない。

### 5.1 PII操作の視覚的強調

患者識別情報に関わる操作が発生する局面では、UIは特別な視覚的処理を施す。

```
PII関連パネルのスタイル:
  背景色: #FFF7ED（オレンジ系薄色）
  左ボーダー: 4px solid #D97706
  ヘッダー: 🔒 アイコン + 「個人識別情報を含む操作」ラベル
  列名表示: オリジナル名は "●●●" マスク → Security Agent承認後に展開ボタン
```

**var_n エイリアス表示ルール:**
- 統計結果・図・原稿のvar_n表示は常に括弧付きで元の概念を表示
  例: `var_1 (収縮期血圧)` — Security Agentが復元した後のみ表示
- Security Agent承認前は `var_1 (---)` と表示し、情報が保護されていることを示す

### 5.2 人間承認ポイントの一貫した表現

「ここで人間が判断しなければ進まない」ポイントは、全画面で統一されたパターンで表現する。

```
承認パネルの構成:
  ┌──────────────────────────────────────────┐
  │  🟣 [HUMAN APPROVAL REQUIRED]            │
  │                                          │
  │  [承認項目の明確な説明]                  │
  │  例: Rスクリプトの実行を承認します。     │
  │      この操作は取り消せません。          │
  │                                          │
  │  □ 内容を確認しました                   │  ← チェックボックス
  │                                          │
  │  [キャンセル]      [承認して実行]        │
  └──────────────────────────────────────────┘

  - 承認ボタンはチェックボックスにチェックが入るまで非活性
  - 承認ボタンは右側に配置（Fittsの法則: 右手優位の操作フロー）
  - 全承認操作は監査ログに記録される（タイムスタンプ・ユーザーアクション）
```

**承認不要な操作との区別:**
- 通常のナビゲーション・表示操作: 承認パネルなし
- Agent実行（自動進行）: 右ペインのAgentアクティビティフィードに記録のみ
- 承認必要操作: 紫色のパネルで必ず明示

### 5.3 セキュリティイベントの通知

`security.yaml`のBREACH / CRITICAL / WARNING イベントは、以下の優先度でUIに伝達する。

| イベント分類 | UI表現 | 操作可否 |
|------------|--------|---------|
| BREACH | 全画面オーバーレイ（赤）「セキュリティ違反が検出されました。作業を中断してください。」| 全操作ブロック |
| CRITICAL | ステータスバー点滅（赤）+ 右ペインに詳細 | 関連操作のみブロック |
| WARNING | ステータスバー（オレンジ）+ 右ペインに通知 | 操作可能 |
| INFO | 右ペインのAgentアクティビティフィードに記録のみ | 操作可能 |

## 6. Offline First対応（AP-012）

### 6.1 接続状態インジケーター

ステータスバー右端に常時表示。

```
● オンライン    : 緑ドット + "オンライン"（外部API利用可能）
● オフライン    : グレードット + "オフライン（コア機能のみ）"
● 同期中        : スピナー + "同期中..."
```

### 6.2 オフライン時のUI制御

CIEのコア機能（統計解析・レポート生成）はオフラインで完全に動作する。オフライン時に制限される機能は明示的にグレーアウトされ、理由が表示される。

| 機能 | オフライン時 |
|------|-------------|
| データアップロード（ローカルファイル） | ✅ 利用可能 |
| 統計解析実行 | ✅ 利用可能 |
| レポート生成 | ✅ 利用可能 |
| Google Drive/Docsへのエクスポート | 🚫 グレーアウト（「オンライン接続が必要です」） |
| 外部APIパッケージ取得 | 🚫 グレーアウト（事前承認済みパッケージのみ利用可能） |

### 6.3 ローカル保存とリカバリー

- すべての実行状態はローカルDBに自動保存（workflow.schema.jsonに準拠）
- アプリケーション再起動時に「未完了のワークフローを再開しますか？」を提示
- オフライン中に完了したレポートはエクスポートキューに保持され、オンライン復帰時に自動処理

## 7. アクセシビリティ・既視感の工夫

### 7.1 アクセシビリティ基準

- **WCAG 2.1 AA準拠** を最低基準とする
- コントラスト比: テキストと背景は4.5:1以上（ダッシュボードの数値表示を含む）
- キーボードナビゲーション: すべての主要操作はマウスなしで完結可能
- スクリーンリーダー対応: WAI-ARIA属性を必須実装。特に承認ダイアログはrole="alertdialog"
- フォーカスインジケーター: 非表示にしない。青色のアウトラインを維持
- 色だけで状態を伝えない: アイコン・テキストラベルを常に併記

### 7.2 既視感の工夫

CIEを初めて使う臨床研究者が「見たことがある」と感じる要素を意図的に組み込む。

| 参照UI | 取り込む要素 | 適用箇所 |
|--------|-------------|---------|
| **RStudio** | 4ペイン構成のメンタルモデル、コンソール出力エリア | 全体レイアウト、実行ログ表示 |
| **VS Code** | サイドバーのアイコンナビ、ステータスバー、コマンドパレット (Ctrl+P) | 左ペインナビ、ステータスバー |
| **REDCap** | フォームのバリデーション表示、ステップ式フォーム、必須項目の赤アスタリスク | 入力フォーム、データ品質チェック |
| **Google Docs** | コメントアノテーション、右マージンのサジェスト表示 | 原稿エディタ、unresolved_items表示 |
| **Jupyter Notebook** | セル単位の実行と出力表示、実行順序番号 | ワークフローステップ、解析結果表示 |

### 7.3 Progressive Disclosure（段階的開示）

CIEは初期状態でシンプルに見せ、詳細は必要に応じて展開する。

```
レベル1（常時表示）:
  - ワークフロー全体の進捗（5ステップ程度に圧縮）
  - 現在のステータスと次のアクション
  - Critical Issueのみ

レベル2（クリックで展開）:
  - 各ステップの詳細（仮定チェック結果、信頼区間等）
  - Agentが選択した手法の根拠
  - 警告レベルの品質問題

レベル3（詳細モード）:
  - Agent間の全入出力ペイロード（JSON表示）
  - 完全な監査ログ
  - var_nエイリアスマッピング（Security Agent承認後）
```

## 8. コンポーネントモデルとの整合

本文書はcomponent-model.mdのPresentation Layer定義と以下の点で完全に整合する。

| component-model.mdの制約 | ui-model.mdでの実装 |
|--------------------------|---------------------|
| UIにビジネスロジックを持たない | 全計算・判断はAgent経由。UIはスキーマ検証済み出力の表示のみ |
| UIから直接Runtimeへのアクセス禁止 | 実行はOrchestratorへのリクエスト経由のみ |
| UIは認証・ファイルアップロード・進捗表示・結果表示の責務のみ | SCR-01〜07はこの責務の範囲内に収まる |
| Agentがワークフローを定義しない | UIはworkflow.yamlの可視化のみ（SCR-03） |

## 付録: 画面遷移の概要
