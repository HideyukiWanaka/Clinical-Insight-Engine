System Requirement Document (SRD) for AI Agent

Target Product: Clinical Insight Engine (CIE)
Role of AI Agent: You are an Expert Full-Stack Developer and Data Engineer. Your task is to build the CIE application strictly following the architecture, schemas, and constraints defined below. Do NOT make assumptions outside of this document.

1. System Architecture & Tech Stack
• Frontend: Next.js (App Router), React, TypeScript, TailwindCSS, Zustand (State Management).
• Backend: Python 3.11+, FastAPI (REST API).
• Statistical Engine: Dockerized R environment (isolated). Communication via REST API (Plumber) or Python-to-R bridge (rpy2).
• AI Service: Latest Native Multimodal LLM (e.g., Claude 3.5 Sonnet or latest equivalent). Capable of single-shot text + image processing.
• Integrations: Google Workspace API (Slides/Docs/Drive), OAuth 2.0.
• Database: PostgreSQL (User profiles, Workflow definitions, Template mappings, Audit logs).
• Storage: Ephemeral local storage or Redis for active session clinical data.

2. Core Data Models (Type Definitions)
All implementations must strictly adhere to these interfaces.

[typescript]
// 1. 解析ワークフローの定義 (Sequence Automation)
interface Workflow {
  id: string;
  userId: string;
  name: string;
  nodes: WorkflowNode[]; // 処理のステップ
  edges: WorkflowEdge[]; // ステップの順序
  createdAt: Date;
  updatedAt: Date;
}

interface WorkflowNode {
  id: string;
  type: 'INPUT' | 'CLEAN' | 'ANALYZE' | 'VISUALIZE' | 'OUTPUT';
  skillId?: string; // 実行するAnalysis SkillのID (例: 'table1_generator', 'logistic_regression')
  parameters: Record<string, any>; // 解析パラメータ (例: x_axis, y_axis, adjust_vars)
}

// 2. テンプレートマッピング定義 (Template Injection)
interface TemplateMapping {
  templateId: string;
  googleDriveFileId: string;
  tags: Array<{
    placeholder: string; // 例: "{{age_mean}}" または "{{km_curve_1}}"
    type: 'TEXT' | 'IMAGE' | 'TABLE';
    dataSourceNodeId: string; // どのWorkflowNodeの結果を紐付けるか
  }>;
}

// 3. ビジュアルリファレンス要求 (Visual Reference)
interface VisualReferencePayload {
  referenceImageBase64: string; // ユーザーがアップロードした参考画像
  targetVariables: {
    x?: string;
    y?: string;
    group?: string;
    [key: string]: string | undefined;
  };
}


3. Core Features & Execution Logic (100% Unambiguous Rules)

Feature 1: Template Injection Engine

• Logic:

    1. Authenticate user via Google Workspace API.

    2. Clone the user-specified Google Slide/Doc template to the user's Drive.

    3. Parse the cloned document for regex pattern \{\{[a-zA-Z0-9_]+\}\}.

    4. Fetch the execution results (DataFrames, Strings, Image paths) from the designated WorkflowNode.

    5. Replace TEXT tags with formatted strings (e.g., P-values rounded to 3 decimal places).

    6. Replace IMAGE tags by inserting base64 encoded PNGs generated by R, preserving the original aspect ratio and position of the placeholder.

Feature 2: Visual Reference (Image-to-Code generation)

• Logic (Single-Shot Multimodal Request):

    1. Receive VisualReferencePayload from the frontend.

    2. Multimodal API Call: Send the image and JSON instructions in a single request to the Native Multimodal LLM.

    3. System Prompt: > "You are an expert R data scientist. Analyze the attached chart image (extract chart type, color hex codes, theme, gridlines, legend position). Generate a completely reproducible R ggplot2 script that mimics this exact design, using the dataframe df and the provided targetVariables. Output ONLY raw R code, no markdown wrapping."

    4. Execute the generated R code directly in the isolated R container.

    5. Return the generated PNG image to the frontend for preview.

Feature 3: Native R Export (Full Reproducibility)

• Constraint: The system MUST NOT generate random scripts post-analysis. It must concatenate the exact code strings executed during the user's session.

• Logic: Generate and allow the user to download a reproducible_script.R file structured as follows:

4. Security & Data Handling (Zero Trust Policy)

• Rule 1 (Data Lifecycle): Clinical data (CSV/Excel) is uploaded to ephemeral storage. It MUST be permanently deleted immediately after the workflow execution finishes or the session times out.
• Rule 2 (LLM Privacy): NEVER send raw clinical data rows to the LLM API. Only send metadata (Column names, Data types: numeric, factor) for code generation or workflow suggestions.
• Rule 3 (Execution Isolation): The R execution container must be completely sandboxed without external internet access to prevent data exfiltration.

5. Error Handling & Edge Cases

• Data Type Mismatch: If a user maps a categorical variable to a node requiring continuous data, return HTTP 400 with a standardized JSON error:
{"error": "TYPE_MISMATCH", "details": "Column 'Age_Group' is factor, expected numeric."}
• Execution Timeout: If an R script execution exceeds 30 seconds, terminate the sub-process, prevent memory leaks, and return HTTP 408 Request Timeout.
• Google API Rate Limits: Implement exponential backoff for Google Drive/Slides API batch updates to avoid Quota Exceeded errors.
• Invalid R Code Output: If the LLM generates invalid R code for the Visual Reference feature, catch the stderr from the R container and retry the LLM request up to 2 times with the error message appended.
