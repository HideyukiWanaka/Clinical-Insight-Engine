## 目的

本文書はCIE Platformのプレゼンテーション層が遵守すべきUI設計原則を定義する。
architecture/ui-model.mdが「何を作るか」を示すのに対して、
本文書は「どのように作るか」の判断基準を提供する。

実装者・レビュアーは本文書を照合基準として使用する。
新しいUI要素を追加する際は、本文書の全原則に照らして設計の妥当性を確認すること。

## UP-001: Intent First表示

**原則:** ユーザーの研究意図（intent_object）は常に最優先で表示される。
AIの処理結果ではなく、ユーザーの言葉と意図がUIの中心に置かれる。

**実装規則:**

1. 研究意図入力エリア（SCR-02）は画面の視覚的重心に配置する。
   最小サイズ: 中央ペインの幅の100%、高さ200px以上。

2. Planner Agentが生成したintent_objectは、ユーザーの入力テキストの
   「翻訳結果」として表示する。翻訳結果がユーザーの意図と異なる場合に
   即座に修正できるUI（インラインエディタ）を提供する。

3. ワークフロー全体を通じて、現在処理中の研究意図サマリーを
   左ペインの最上部に常時表示する。

4. AIが提案した内容（手法選択・テキスト草稿）には必ず根拠を付記する。
   根拠なしの提案UIは実装してはならない。

**禁止事項:**
- ユーザーが入力前にAIが先行して何かを表示すること
- intent_objectの自動確定（必ず人間が確認・承認する）

## UP-002: Human Authority可視化

**原則:** 人間が判断・承認すべき操作は、他のすべての操作から視覚的に区別される。
「AIが自動で進んでいる」と「人間が行動しなければならない」は
一目で判別できなければならない。

**実装規則:**

1. **承認待ち状態の配色:** `--cie-approval` (#7C3AED、紫) を一貫して使用する。
   この色はシステム内でHuman Authority専用とし、他の目的に転用しない。

2. **承認パネルの構成要素（必須）:**
   - 紫ヘッダー: 🟣 HUMAN APPROVAL REQUIRED
   - 承認対象の明確な説明（1〜2文）
   - 操作の不可逆性の明示（「この操作は取り消せません」等）
   - チェックボックス: 「内容を確認しました」（未チェック時はボタン非活性）
   - 二択ボタン: [キャンセル] [承認して実行]（承認ボタンは右配置）

3. **承認ボタンの二段階保護:**
   チェックボックス入力 → ボタン活性化 → クリック → 確認ダイアログ → 実行
   一度のクリックで実行されてはならない。

4. **承認操作のフィードバック:**
   承認完了後、右ペインに「承認済み（タイムスタンプ）」を記録して表示する。

5. **承認不要操作との明確な区別:**
   通常のナビゲーション・表示切替・設定変更は承認パネルを表示しない。
   判断が難しい場合は「この操作に承認が必要か」をpermissions.yamlで確認する。

**禁止事項:**
- 承認パネルを自動閉鎖・タイムアウトで処理すること
- 承認待ちタスクを右ペインから折りたたむこと（承認済みになるまで）
- 承認ボタンに緑・赤以外の色（紫以外の色）を使用すること

## UP-003: Progressive Disclosure

**原則:** UIは初期状態でシンプルであり、詳細は必要に応じて段階的に展開する。
臨床研究者の主要ジャーニーを妨げる情報は、デフォルトで非表示とする。

**3段階の情報階層:**

```
レベル1（デフォルト表示）
  目的: 「今何が起きているか」「次に何をすべきか」の把握
  内容:
    - ワークフロー全体の進捗（ステップ名と状態のみ）
    - 現在のステータス
    - Critical Issueのみ（Warningはバッジ数のみ）
    - 承認待ちアクション

レベル2（展開で表示）
  目的: 「なぜこうなったか」「詳細はどうか」の確認
  トリガー: 各ステップのクリック、「詳細を表示」リンク
  内容:
    - 仮定チェック結果の全項目
    - Agent選択の根拠
    - Warning一覧
    - 各変数の統計サマリー

レベル3（詳細モード）
  目的: 開発者・上級ユーザーによる深い検証
  トリガー: 「詳細モード」トグル、Ctrl+Shift+D
  内容:
    - Agent間のJSON入出力ペイロード
    - 完全な監査ログ（全イベント）
    - var_nエイリアスマッピング詳細
    - Rスクリプト内容のプレビュー
```

**実装規則:**

1. 各コンポーネントはレベル1の状態をデフォルトレンダリングとする。
   レベル2・3は遅延ロード（Lazy Load）で実装する。

2. 展開・折りたたみのトグルは常に同じ場所（コンポーネント右上）に配置する。

3. 展開状態はセッション中に記憶する（再ロード時にリセットしない）。

4. Critical Issueのみレベル3であっても強制展開される。

**禁止事項:**
- 初期表示でレベル2・3の情報を表示すること
- 「詳細を非表示」操作でCritical Issueを隠すこと

## UP-004: Security By Design表示

**原則:** セキュリティ状態とPII操作は、常に視覚的に明確である。
セキュリティが「見えない」状態は設計上の欠陥とみなす。

**実装規則:**

### PII操作パネルのスタイル規則

```css
.cie-pii-panel {
  background-color: #FFF7ED;   /* オレンジ系薄色 */
  border-left: 4px solid #D97706;
  padding: 12px 16px;
}
.cie-pii-panel-header::before {
  content: "🔒 ";
}
```

PII操作パネルが使用される箇所:
- var_nエイリアスとオリジナル列名の対応表示
- dataset.read_rawパーミッション要求時
- r_code.restore_variables実行前後

### セキュリティイベント表示規則

| イベント | ステータスバー | 右ペイン | 中央ペイン |
|---------|-------------|---------|-----------|
| INFO | 変化なし | フィードに追記 | 変化なし |
| WARNING | オレンジドット + カウント | 展開して表示 | 変化なし |
| CRITICAL | 赤点滅 | 強制展開 + 赤ハイライト | 関連操作をブロック |
| BREACH | 赤全体点滅 | — | 全画面オーバーレイ |

### BREACH時の全画面オーバーレイ仕様

```
背景: rgba(220, 38, 38, 0.95) — ほぼ不透明な赤
表示内容:
  🚨 セキュリティ違反が検出されました
  
  実行中のワークフローを中断しました。
  監査ログを確認し、管理者に報告してください。
  
  エラーコード: [BREACH_EVENT_CODE]
  発生時刻: [ISO8601 timestamp]
  
  [監査ログを表示] [アプリを再起動]

操作: 上記2ボタン以外の全操作をブロック
```

**禁止事項:**
- セキュリティイベントの自動消去（ユーザーが明示的に確認するまで表示を維持）
- PII操作パネルのスタイルを他用途に使用すること
- BREACHオーバーレイを閉じるボタンを提供すること

## UP-005: Observable Execution

**原則:** すべてのAgent実行は観察可能でなければならない。
「AIが何をしているかわからない」状態は許容されない。AP-018の視覚的実装。

**実装規則:**

1. **Agentアクティビティフィード（右ペイン常時表示）:**
   各Agentの起動・完了・エラーをリアルタイムでストリーミング表示する。
   表示形式: `[タイムスタンプ] [Agent名] → [アクション]: [結果サマリー]`
   例: `14:32:07  statistics → method_selected: welch_t_test (normality_passed=true)`

2. **ステップ詳細モーダル（SCR-03）:**
   各ワークフローステップをクリックすると、そのステップの
   入力ペイロード（input）と出力ペイロード（output）をJSONビューアで表示する。
   ペイロードはschema_refを表示し、スキーマとの対応を確認できる。

3. **実行中のスピナー表示:**
   Agentが処理中の場合、該当ステップに回転アニメーションを表示する。
   スピナーの横に「処理中... (XX秒経過)」を表示し、タイムアウトが近づくと
   黄色に変化する（残り30秒でwarning、残り10秒でcritical）。

4. **意思決定の根拠表示（Decision Trace）:**
   decision_branch_takenが記録されているステップには「🔀 判断あり」バッジを表示。
   クリックで判断内容（どの分岐を選んだか・理由）をモーダルで表示する。

**禁止事項:**
- Agentの処理をUIが「まとめて完了」として表示すること（全ステップを個別に表示）
- 失敗したステップを後から成功として表示しないこと

## UP-006: Offline First対応

**原則:** ネットワーク接続の有無にかかわらず、コア機能は完全に動作する。
接続状態はユーザーに常に明示され、オフライン時の制限は事前に案内される。

**実装規則:**

1. **接続状態インジケーターの常時表示:**
   ステータスバー左端に配置。状態変化時はアニメーションでフラッシュする。
   ```
   ● オンライン    : #059669 (緑) + "オンライン"
   ● オフライン    : #4B5563 (グレー) + "オフライン"
   ● 接続確認中    : スピナー + "確認中..."
   ```

2. **オフライン制限の事前案内:**
   接続がオフラインに変わった瞬間、非活性化される機能のボタンに
   ツールチップで「この機能はオンライン接続が必要です」を表示する。
   非活性化はCSSのdisabled属性で実装し、クリックを物理的に防止する。

3. **オフライン中の作業継続:**
   ローカルで実行可能な全操作（統計解析・レポート生成・監査ログ閲覧）は
   オフライン状態でも通常通り使用できる。
   保存はすべてローカルDBへのオートセーブで行われる。

4. **オンライン復帰時の処理:**
   エクスポートキューに保留されたタスクがある場合、復帰後に
   「オンラインに復帰しました。保留中の1件のエクスポートを処理しますか？」
   のトースト通知を表示する。

**禁止事項:**
- ネットワーク状態の変化をユーザーに通知せずに処理すること
- オフライン時にコア機能を無効化すること

## UP-007: 既視感と学習コスト最小化

**原則:** 臨床研究者が既に使い慣れているUIパターンを積極的に採用する。
独自性より親しみやすさを優先する。

**実装規則:**

1. **キーボードショートカットの標準化:**

   | ショートカット | 動作 | 参照 |
   |-------------|------|------|
   | `Ctrl+B` | 左ペイン折りたたみ | VS Code |
   | `Ctrl+P` | コマンドパレット | VS Code |
   | `Ctrl+Enter` | 現在のステップを実行 | Jupyter |
   | `Ctrl+Z` | 入力を元に戻す | 標準 |
   | `Ctrl+S` | 手動保存（通常はオートセーブ） | 標準 |
   | `Ctrl+Shift+D` | 詳細モード切替 | CIE独自 |
   | `Escape` | モーダルを閉じる | 標準 |

2. **ドラッグ&ドロップ対応:**
   データセットアップロードエリアはドラッグ&ドロップを受け付ける。
   ドロップ中はエリアの背景色を変化させてフィードバックを提供する。

3. **コマンドパレット（Ctrl+P）:**
   VS Code風の検索インターフェース。以下の操作を検索可能とする:
   - 画面遷移（「データ品質」「統計設定」等）
   - 最近のプロジェクト
   - ヘルプ・ドキュメント検索

4. **ツールチップの一貫した表示:**
   すべてのアイコンボタン・略語・統計用語にツールチップを提供する。
   表示遅延: 500ms（誤表示防止）
   統計用語の場合は日本語説明 + 英語正式名を併記する。

**禁止事項:**
- 標準的なキーボードショートカット（Ctrl+Z, Ctrl+C等）の動作を上書きすること
- カスタムスクロール実装による標準スクロール動作の破壊

## UP-008: エラーの明確な伝達

**原則:** エラーメッセージは「何が起きたか」と「何をすればよいか」を必ず伝える。
技術的エラーコードはユーザーから隠蔽しない（折りたたんで提供する）。

**エラーメッセージの構成:**

```
[アイコン] [タイトル（何が起きたか）]

[説明文（平易な日本語で1〜2文）]

[推奨アクション（具体的な操作手順）]

▼ 技術的詳細（折りたたみ）
  エラーコード: INTENT_EXTRACTION_FAILED
  エージェント: planner
  タイムスタンプ: 2025-06-27T14:32:07Z
```

**エラー種別ごとのUIスタイル:**

| 種別 | アイコン | 色 | 表示場所 |
|------|---------|-----|---------|
| Validation Error | ⚠️ | `--cie-warning` | フォームフィールド直下 |
| Agent Error (RECOVERABLE) | 🔄 | `--cie-warning` | 右ペイン + 該当ステップ |
| Agent Error (UNRECOVERABLE) | ❌ | `--cie-critical` | 中央ペイン + 右ペイン |
| Security Event | 🔒 | `--cie-critical` | ステータスバー + 右ペイン |
| Network Error | 📡 | `--cie-gray-600` | ステータスバー |

**実装規則:**

1. エラーメッセージに技術的略語をそのまま表示しない。
   例: `QUALITY_GATE_BLOCKED` → 「データ品質チェックで問題が検出されました」

2. RECOVERABLE エラーには「再試行」ボタンを提供する。

3. エラーは監査ログに自動記録される（UIが個別に記録する必要はない）。

**禁止事項:**
- エラーの自動消去・タイムアウト消去（ユーザーが明示的に確認するまで表示）
- 「エラーが発生しました」のみの説明不足メッセージ

## UP-009: 一貫した状態表現

**原則:** 同じ状態は、システム内のすべての場所で同じ視覚表現を使用する。
状態の視覚的定義はこの原則が唯一の源泉である。

**ワークフローステップの状態定義:**

| 状態 | アイコン | 色 | 説明 |
|------|---------|-----|------|
| `pending` | ○ | `--cie-gray-200` | 未実行 |
| `running` | ⟳ (回転) | `--cie-blue-500` | 実行中 |
| `completed` | ✓ | `--cie-success` | 正常完了 |
| `failed` | ✕ | `--cie-critical` | 失敗 |
| `waiting_for_human` | 🟣 | `--cie-approval` | 承認待ち |
| `retrying` | ↻ | `--cie-warning` | リトライ中 |
| `skipped` | — | `--cie-gray-400` | スキップ済み |

これらの状態定義は orchestrator.yaml の `valid_states` と完全に対応する。

**品質評価スコアの表示基準:**

| スコア範囲 | 表示色 | バッジラベル |
|-----------|--------|------------|
| 90〜100 | `--cie-success` | PASS |
| 70〜89 | `--cie-warning` | REVIEW |
| 0〜69 | `--cie-critical` | FAIL |
| 未評価 | `--cie-gray-400` | — |

**禁止事項:**
- 同じ状態に異なる色・アイコンを使用すること
- 状態定義をコンポーネントごとに独自拡張すること

## UP-010: アクセシビリティ必須

**原則:** アクセシビリティはオプション機能ではなく、設計の必須要件である。
WCAG 2.1 AA準拠をすべてのコンポーネントに対して保証する。

**必須実装事項:**

1. **コントラスト比:**
   - 通常テキスト（14px以下）: 4.5:1以上
   - 大きいテキスト（18px以上またはBold 14px以上）: 3:1以上
   - UI要素の境界線: 3:1以上

2. **キーボードナビゲーション:**
   - Tabキーで全インタラクティブ要素をフォーカス可能
   - フォーカスインジケーターを非表示にしない
   - モーダルダイアログはフォーカストラップを実装する（Escで閉じる）

3. **スクリーンリーダー対応（必須WAI-ARIA）:**
   - 承認ダイアログ: `role="alertdialog"` `aria-modal="true"`
   - 実行中スピナー: `role="status"` `aria-live="polite"`
   - エラーメッセージ: `role="alert"` `aria-live="assertive"`
   - ステップ状態変化: `aria-live="polite"`

4. **色だけで状態を伝えない:**
   すべての色による状態表現にアイコンまたはテキストラベルを必ず併記する。

5. **フォームの入力支援:**
   - すべてのinput要素にlabel要素を関連付ける
   - エラーメッセージはinput要素の`aria-describedby`で関連付ける
   - 必須フィールドは`aria-required="true"`を設定する

**アクセシビリティチェックリスト（PR前確認):**
- [ ] axe-core / Lighthouse Accessibility スコア 90以上
- [ ] キーボードのみで全主要操作を完了できる
- [ ] 色を白黒に変換しても状態が判別できる
- [ ] NVDA/VoiceOverで主要画面を音声確認済み
