## Context

### 問題の所在

CIE Platformにおいて、PlannerAgentとOrchestratorの責務境界に
以下の曖昧さが存在していた。

**曖昧さ1 — ワークフロー選択の主体が未定義**

`planner.yaml`は`workflow_definition_mutation`を`strictly_forbidden`としているが、
「ワークフロー定義の変更」と「ワークフロー定義の選択」の区別が明文化されていなかった。
Plannerがintent_objectを生成した後、誰がどのworkflowを起動するかが不明瞭だった。

**曖昧さ2 — ワークフローの静的/動的性が未定義**

`spec/workflow.yaml`は`clinical_analysis`という単一ワークフローのみを定義しているが、
研究デザイン（RCT・観察研究・生存解析・メタ解析）によって実際のDAGノード構成は
異なるべきである。PlannerがDAGを動的生成するのか、Orchestratorが静的定義から選択するのかが
不明だった。

**曖昧さ3 — PROJECT_RULES.mdとの整合**

PROJECT_RULES.md section 3.2が「Agents execute workflows. Agents never define workflows.」と
定めているため、Plannerが何らかの形でワークフロー構成に関与することが
この原則と矛盾するかどうかが不明だった。

---

## Decision

### 採用する設計: 静的マルチワークフロー + Orchestratorによる選択

以下を正式な設計として確定する。

#### 原則1 — ワークフロー定義は完全に静的

すべてのワークフローDAGは`spec/workflow.yaml`にアーキテクチャとして事前定義される。
Plannerを含むいかなるAgentも、実行時にDAGノードを追加・削除・変更してはならない。
これはPROJECT_RULES.md section 3.2の「Agents never define workflows」の
文字通りの実装である。

#### 原則2 — Plannerの出力はintent_objectのみ

Plannerの唯一の責務は、自然言語プロンプトを`analysis-request.schema.json`に
準拠した**intent_object**に変換することである。
Plannerはワークフローを「選択」も「推薦」もしない。
intent_objectには`workflow_id`フィールドを含まない。

#### 原則3 — ワークフロー選択はOrchestratorの責務

Orchestratorは受け取ったintent_objectを分析し、
`spec/workflow.yaml`の`workflow_registry`に登録された
ワークフロー定義の中から最適なものを**決定論的ルール**に基づいて選択する。
この選択ロジックはOrchestratorに実装され、固定ルールテーブルとして管理される。

#### 原則4 — ワークフロー数は有限の静的セット

初期実装では以下の4ワークフローを定義する。

| workflow_id | 対象 | 判定条件（Orchestrator使用）|
|------------|------|--------------------------|
| `clinical_analysis_standard` | 標準的な群間比較・相関・回帰 | `objective ∈ {between_group_comparison, correlation_analysis, regression_analysis}` |
| `clinical_analysis_survival` | 生存解析 | `outcome_type = "survival"` |
| `clinical_analysis_meta` | メタ解析 | `objective = "systematic_review"` |
| `clinical_analysis_prediction` | 予測モデル開発・検証 | `objective = "prediction_model"` |

将来の追加ワークフローはADRを経て`spec/workflow.yaml`に追記される。

## Responsibility Boundary（確定版）

```
ユーザーの自然言語
       │
       ▼
┌──────────────────────────────────────────────────────────┐
│  Planner Agent の責務（ここまで）                        │
│                                                          │
│  ・自然言語 → intent_object への変換                    │
│  ・objective, outcome_type, predictor_type の特定       │
│  ・paired, subject_id_var, n_groups_estimate の推定     │
│  ・曖昧さの検出と clarification_options の生成          │
│                                                          │
│  ✕ workflow_id の決定                                   │
│  ✕ DAGノードの選択・変更                                │
│  ✕ 統計手法の選択（Statistics Agentの責務）             │
└──────────────────────┬───────────────────────────────────┘
                       │ intent_object（analysis-request.schema.json）
                       ▼
┌──────────────────────────────────────────────────────────┐
│  Orchestrator の責務（ここから）                         │
│                                                          │
│  ・intent_object を受け取る                             │
│  ・workflow_selection_rules に基づきworkflow_idを決定    │
│  ・spec/workflow.yaml から対応DAGを読み込む             │
│  ・task_dispatch_loop でDAGノードを順次実行             │
│  ・state_machine の管理                                  │
│  ・capability_token の発行・失効                         │
│  ・audit_log への記録                                    │
│                                                          │
│  ✕ intent_objectの内容を変更すること                    │
│  ✕ ワークフロー定義を動的に生成・変更すること           │
│  ✕ 統計手法の選択                                       │
└──────────────────────────────────────────────────────────┘
```

## Workflow Selection Rules（Orchestrator実装）

```yaml
# Orchestrator内部の固定ルールテーブル（変更にはADRが必要）
workflow_selection_rules:

  - rule_id: "WS-001"
    priority: 1
    condition:
      outcome_type: "survival"
    workflow_id: "clinical_analysis_survival"

  - rule_id: "WS-002"
    priority: 2
    condition:
      objective: "systematic_review"
    workflow_id: "clinical_analysis_meta"

  - rule_id: "WS-003"
    priority: 3
    condition:
      objective: "prediction_model"
    workflow_id: "clinical_analysis_prediction"

  - rule_id: "WS-004"
    priority: 99   # default fallback
    condition: {}   # 他のルールに該当しない場合
    workflow_id: "clinical_analysis_standard"

  selection_policy:
    deterministic: true          # 同一intent_objectに対して常に同一workflow_idを返す
    on_no_match: "use_default"   # WS-004を適用
    on_requires_human_clarification_true:
      action: "suspend"          # intent_objectが未確定の場合はworkflow選択を保留
      resume_after: "clarification_resolved"
```

## Consequences

### ポジティブな影響

1. **責務の明確化:** PlannerとOrchestratorの境界が1行で説明できる。
   「PlannerはWHATを決める。OrchestratorはHOWを実行する。」

2. **テスタビリティの向上:** Planner単体テストはworkflow知識なしに実施可能。
   ワークフロー選択ルールはOrchestratorの単体テストとして独立して検証可能。

3. **PROJECT_RULES.md完全準拠:** Plannerはワークフローを「選択」も「定義」もしないため、
   section 3.2「Agents never define workflows」と矛盾しない。

4. **Offline First維持:** ワークフロー選択が静的ルールテーブルで行われるため、
   外部サービスへの依存なしに動作する。

### トレードオフ

1. **柔軟性の制約:** 新しい研究デザインへの対応には`spec/workflow.yaml`の更新と
   ADRの作成が必要。ただしこの制約はPROJECT_RULES.mdの設計原則に沿っている。

2. **ワークフロー数の管理:** 将来的にワークフロー数が増加した場合、
   WS-ruleの優先度管理が複雑になる可能性がある。
   この問題が顕在化した場合は別途ADRで対処する。
