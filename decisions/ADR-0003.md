## Context

### 問題の所在

CIE PlatformのPhase 6（Orchestrator・R実行エンジン）完成時点で、
`knowledge/` は以下の課題を抱えていた。

**課題1 — 知識ベースが完全に静的**

現行の `knowledge/` は初期構築時にのみ更新され、
学会統計ガイドライン（CONSORT, ICH-E9等）の改訂や
施設固有の解析方針・プロトコルをシステムに組み込む手段が存在しない。
研究者は都度LLMプロンプトに文脈を手動で追記する必要があり、
再現性・一貫性の面で重大な課題となっていた。

**課題2 — 知識更新フローの欠如**

evaluation/ が生成した知識品質の問題を `knowledge/` に
フィードバックする経路が存在しない。
ADR-0002がSkillに対して解決したのと同じ問題が
Knowledgeには未解決のまま残っていた。

**課題3 — ライセンスリスクの潜在**

PDFなどのドキュメント処理ライブラリを直接実装に埋め込むと、
将来のSaaS化（Phase 5）でAGPL等のライセンス汚染リスクが顕在化する。

**課題4 — 管理ワークフローと解析ワークフローの混在リスク**

知識アップロードのような「管理タスク」を
解析用の WorkflowRegistry（ADR-0001のWS-001〜WS-004）に
混在させると、Plannerが管理タスクを誤って選択する危険性がある。

---

## Decision

### 採用する設計: Knowledge Ingestion Pipeline (KIP)

以下を正式な設計として確定する。

---

#### 原則1 — knowledge/ の3ネームスペース構造（ADR-0002と対称）

ADR-0002の `skills/` 3ネームスペース構造と完全に対称な形で
`knowledge/` を拡張する。

```
knowledge/
├── official/           ← CIE公式の参照知識（現行の knowledge/ を移管）
│   ├── statistics/     ← ICH-E9, CONSORT等の解釈済み知識
│   ├── clinical/
│   ├── reporting/
│   ├── R/
│   ├── Python/
│   └── visualization/
│
├── institutional/      ← 施設・プロジェクト固有の承認済み知識
│   ├── REGISTRY.yaml   ← 登録済み知識エントリの台帳（sources of truth）
│   └── {entry_id}/
│       ├── KNOWLEDGE.md    ← 抽出・構造化された知識本体
│       ├── METADATA.yaml   ← ソース情報・承認記録・バージョン・有効期限
│       ├── source/         ← 原典ファイル（PDF等）の不変アーカイブ
│       └── versions/       ← 論理削除・改訂による旧バージョンのアーカイブ
│
└── pending/            ← 審査待ちの抽出済み知識（一時領域）
    └── {entry_id}/
        ├── EXTRACTED.md       ← AI抽出結果（未承認・参照専用）
        ├── SOURCE_HASH.txt    ← 原典ファイルのSHA-256（重複検出用）
        └── REVIEW_REQUEST.yaml
```

| 名前空間 | 対応するskills/ | 更新主体 | ガバナンス |
|---------|---------------|---------|-----------|
| official | core/ | CIEチーム | SkillLifecycleと同等のプロセス + 人間承認 |
| institutional | user/ | 研究者・Admin | 人間承認 + REGISTRY.yaml |
| pending | —（一時領域）| AI | 自動（承認前の一時保管のみ） |

#### 原則2 — official/ の不変性維持

`knowledge/official/` 配下はユーザーが直接変更できない。
現行の PROJECT_RULES.md Section 12 の適用範囲を
`knowledge/official/` に明示的に限定する。
（従来の "Knowledge is immutable" はこのネームスペースを指す）

#### 原則3 — AbstractDocumentParser によるライブラリ依存の完全隠蔽

ドキュメント変換ライブラリ（当面: PyMuPDF / pymupdf4llm）への依存は、
必ず抽象基底クラス `AbstractDocumentParser` を介してのみ行う。

```python
# cie/knowledge/parsers/base.py
from abc import ABC, abstractmethod
from dataclasses import dataclass
from pathlib import Path

@dataclass
class ParsedDocument:
    raw_text: str
    structured_markdown: str
    page_count: int
    source_hash: str       # SHA-256 of original bytes
    parser_name: str       # "pymupdf4llm_v0.0.17" 等（再現性のため記録）
    parser_version: str

class AbstractDocumentParser(ABC):
    """
    ドキュメント変換ライブラリのインターフェース。
    実装クラスを差し替えても KnowledgeIngestionAgent の変更は不要。
    """

    @abstractmethod
    def can_parse(self, suffix: str) -> bool:
        """このパーサーが処理可能なファイル拡張子かを返す。"""
        ...

    @abstractmethod
    def parse(self, file_path: Path, file_bytes: bytes) -> ParsedDocument:
        """ファイルをパースして ParsedDocument を返す。"""
        ...
```

```python
# cie/knowledge/parsers/pymupdf_parser.py
class PyMuPDFParser(AbstractDocumentParser):
    """
    PyMuPDF (AGPL-3.0) の実装。
    将来ライセンス変更が必要になった場合はこのクラスのみ差し替える。
    KnowledgeIngestionAgent は AbstractDocumentParser しか参照しない。
    """
    def can_parse(self, suffix: str) -> bool:
        return suffix.lower() in {".pdf"}

    def parse(self, file_path: Path, file_bytes: bytes) -> ParsedDocument:
        import pymupdf4llm  # ここだけに局所化
        ...
```

KnowledgeIngestionAgentは `AbstractDocumentParser` のリストを
依存注入で受け取り、ライブラリ名を直接参照しない。

#### 原則4 — Soft Delete（論理削除）による後方互換性の完全保証

institutional/ の削除は「物理削除禁止・論理削除のみ」とする。

**削除フロー:**

```
研究者/Admin がUIで「削除」を選択
       │
       ▼
権限チェック:
  (current_user.id == entry.created_by) OR (current_user.role == "admin")
       │ 条件不成立 → PermissionDeniedError
       │ 条件成立
       ▼
KnowledgeLifecycleService.archive_entry() を呼び出す:
  ① METADATA.yaml の status を "active" → "archived" に変更
  ② archived_at / archived_by を記録
  ③ {entry_id}/ を institutional/{entry_id}/versions/{version}/ へ退避
  ④ institutional/REGISTRY.yaml の当該エントリを status: archived に更新
  ⑤ AuditLog に KnowledgeArchivedEvent を記録
       │
       ▼
過去のworkflow実行の再現時:
  FrozenKnowledgeSet が execution_id 時点のスナップショットを参照
  → archived 後も過去の解析は完全に再現可能
```

物理削除（ファイルの完全消去）は `KnowledgeLifecycleService` においても
実装しない。実装すること自体を `strictly_forbidden` とする。

#### 原則5 — SystemWorkflowRegistry の新設（ADR-0001の死守）

解析ワークフロー（WS-001〜WS-004）と管理ワークフローを完全に分離する。

```
ワークフロー種別の分類:

  AnalysisWorkflowRegistry  （既存: spec/workflow.yaml）
  └── clinical_analysis_standard      [WS-001〜WS-004で選択]
  └── clinical_analysis_survival
  └── clinical_analysis_meta
  └── clinical_analysis_prediction
      ↑
      Planner Agent → Orchestrator の経路で起動

  SystemWorkflowRegistry    （新設: spec/system-workflow.yaml）
  └── knowledge_ingestion             [本ADR]
  └── skill_lifecycle                 [ADR-0002の正式化]
  └── (将来) protocol_generation
  └── (将来) audit_export
      ↑
      UI Event → Orchestrator 直接キック（Plannerを経由しない）
```

**SystemWorkflowRegistryのルール:**
- Planner Agent は SystemWorkflowRegistry を参照しない
- SystemWorkflowRegistry への workflow_id 追加には ADR が必要（ADR-0001準拠）
- Orchestratorは起動経路（Planner経由 or UI直接）でRegistryを切り替える

#### 原則6 — バージョン衝突時の共存方式（上書き禁止）

同一ドキュメントの改訂版がアップロードされた場合、旧版を上書きしない。

```yaml
# KE-0045（改訂版）の METADATA.yaml
entry_id: "KE-0045"
domain: "statistics"
version: "1.0.0"
status: "active"
trust_level: "regulatory"

# 医療文献の親子関係を追跡
related_entries:
  - entry_id: "KE-0012"
    relationship: "supersedes"    # KE-0045 は KE-0012 の改訂版

# KE-0012（旧版）の METADATA.yaml には自動で追記
related_entries:
  - entry_id: "KE-0045"
    relationship: "superseded_by"  # KE-0012 は KE-0045 に置き換えられた
```

UIは `superseded_by` が存在するエントリに対して
⚠️ 「この知識には新しいバージョンがあります」と自動警告を表示する。
旧版の参照は許可するが、新規解析では新版が優先されるよう
FrozenKnowledgeSet の読み込みロジックで制御する。

#### 原則7 — expires_at による最小コスト有効期限アラート

```yaml
# METADATA.yaml のスキーマ追加
expires_at: "2027-12-31"    # ISO-8601 date | null
                             # null = 有効期限なし
```

**UIアラートの実装（バッチ処理なし）:**

```python
# cie/knowledge/loader.py
def check_expiry_warnings(self, entries: list[KnowledgeEntry]) -> list[ExpiryWarning]:
    """
    UIロード時に呼び出し。重いバッチ処理は作らない。
    """
    today = date.today()
    warnings = []
    for entry in entries:
        if entry.expires_at is None:
            continue
        if entry.expires_at < today:
            warnings.append(ExpiryWarning(
                entry_id=entry.entry_id,
                level="expired",
                message=f"⚠️ この知識は期限切れです（{entry.expires_at}）。"
                        "最新ガイドラインを確認してください。"
            ))
        elif (entry.expires_at - today).days <= 90:
            warnings.append(ExpiryWarning(
                entry_id=entry.entry_id,
                level="expiring_soon",
                message=f"📅 この知識はまもなく期限切れになります（{entry.expires_at}）。"
            ))
    return warnings
```

## Knowledge Ingestion Pipeline（KIP）フロー

```
[Phase 1: アップロードと検疫]
─────────────────────────────────────────────────────
  研究者 が UI からドキュメントをアップロード
       │
       ▼
  SystemWorkflowRegistry が knowledge_ingestion を起動
  （Planner を経由しない）
       │
       ▼
  IngestionGuard による5段階チェック:
  ① ファイル形式チェック（許可リスト: .pdf, .md, .docx, .txt）
  ② ファイルサイズ上限チェック（50MB）
  ③ SHA-256 ハッシュ生成・重複アップロード検出
  ④ PDF 内埋め込みスクリプト（JavaScript等）の検出・拒否
  ⑤ 既存 PII 検出エンジンによる患者情報混入チェック
       │ 検査通過
       ▼
  source/ に保管 + AuditLog に DocumentUploadEvent 記録

[Phase 2: AI知識抽出]
─────────────────────────────────────────────────────
  KnowledgeIngestionAgent が起動
  → AbstractDocumentParser.parse() でテキスト化
  → meta/knowledge-extractor/ SKILL で知識を構造化:

    EXTRACTED.md の構造:
    ├── source_info      ← タイトル・著者・発行年・DOI（必須）
    ├── domain_tags      ← [statistics, clinical, reporting, ...]
    ├── knowledge_entries
    │   ├── id: "KE-XXXX"
    │   ├── statement    ← 知識の主張
    │   ├── direct_quote ← 原典からの直接引用（根拠）
    │   ├── confidence   ← AI確信度スコア
    │   └── caveats      ← 限界・注意事項
    ├── trust_level      ← regulatory|peer_reviewed|institutional|experimental
    └── limitations      ← AIが判断できなかった箇所

  → knowledge/pending/{entry_id}/ に一時保存
  → AuditLog に KnowledgeExtractionEvent 記録

[Phase 3: Human Review & Approval]  ← 最重要防衛線
─────────────────────────────────────────────────────
  UI に KnowledgeEntryDraft を提示:
  ├─ [原典を参照]     → source/ の元ファイルを表示
  ├─ [抽出内容を確認]  → EXTRACTED.md のプレビュー
  ├─ [confidence確認] → 低スコアエントリを色分けハイライト
  └─ [関連エントリ確認] → supersedes/superseded_by の表示

  Human が判断:
  ├─ 却下 → pending/ から削除、AuditLog に記録
  ├─ 修正 → EXTRACTED.md を直接編集して再提出
  └─ 承認 → Phase 4へ

  ★ KnowledgeLifecycleService 以外のコードパスで
    institutional/ へ書き込む経路は実装しない

[Phase 4: 知識の統合とバージョニング]
─────────────────────────────────────────────────────
  KnowledgeLifecycleService.register_knowledge() が:
  ① entry_id を採番（KE-XXXX 形式）
  ② pending/ の旧バージョンを versions/ にアーカイブ
  ③ EXTRACTED.md → KNOWLEDGE.md にプロモート
  ④ METADATA.yaml の approved_by / approved_at を記録
  ⑤ related_entries の親子関係を双方向に設定
  ⑥ institutional/REGISTRY.yaml に新エントリを追記
  ⑦ AuditLog に KnowledgeRegistrationEvent 記録
  ⑧ pending/ の一時ファイルを削除

[Phase 5: 実行時の知識利用]
─────────────────────────────────────────────────────
  Orchestrator が workflow 開始時に一度だけ:
  → official/ と institutional/（status: active のみ）をロード
  → superseded_by が存在する場合は新版を優先
  → FrozenKnowledgeSet（immutable tuple）として返す
  → PROJECT_RULES.md Section 12 の "immutable during execution" を保証
  → 実行中の reload() 呼び出しは例外を送出
```

## セキュリティ防衛線（5層）

### Layer 1: 入口検疫（IngestionGuard）

```python
# cie/knowledge/ingestion_guard.py
ALLOWED_EXTENSIONS = {".pdf", ".md", ".txt", ".docx"}
MAX_FILE_SIZE_BYTES = 50 * 1024 * 1024  # 50MB

class IngestionGuard:
    """
    ドキュメントアップロードの検疫。
    全チェック通過でなければ Phase 2 に進まない。
    """
    def inspect(self, file_path: Path, file_bytes: bytes) -> InspectionResult:
        checks = [
            self._check_extension(file_path),
            self._check_file_size(file_bytes),
            self._check_sha256_duplicate(file_bytes),
            self._scan_for_embedded_scripts(file_bytes),  # PDF内JS等
            self._pii_scan_on_raw_content(file_bytes),
        ]
        if any(not c.passed for c in checks):
            raise IngestionError(
                failed_checks=[c for c in checks if not c.passed]
            )
        return InspectionResult(sha256=self._sha256(file_bytes), checks=checks)
```

### Layer 2: PII汚染防止

既存の `PIIFilter`（Phase 2実装済み）を KnowledgeIngestionAgent 内で再利用。
患者識別情報が検出された場合は `PII_DETECTED_IN_DOCUMENT` エラーで拒否。

### Layer 3: 正確性の構造的担保

`source_info.doi` or `source_info.url` のいずれかを必須とし、
出典追跡不能な知識の登録を schema validation で拒否する。

### Layer 4: Trust Level による権威性の格付け

| trust_level | 意味 | UIでの表示 |
|-------------|------|-----------|
| `regulatory` | 規制文書（ICH-E9等） | 🟢 規制文書 |
| `peer_reviewed` | 査読済み論文 | 🔵 査読済み |
| `institutional` | 施設固有の慣行 | 🟡 施設固有 |
| `experimental` | 実験的・暫定的 | 🔴 実験的（要注意） |

### Layer 5: 実行時 Immutability の保証

```python
# cie/knowledge/loader.py
@dataclass(frozen=True)
class FrozenKnowledgeSet:
    """
    PROJECT_RULES.md Section 12:
    "Knowledge is immutable during execution" の実装。
    frozen=True により属性変更はTypeErrorで拒否される。
    """
    loaded_at: datetime
    execution_id: str
    entries: tuple[KnowledgeEntry, ...]  # immutable tuple

    def reload(self) -> None:
        # 実行中の再読み込みは禁止
        raise KnowledgeError("KNOWLEDGE_RELOAD_DURING_EXECUTION_FORBIDDEN")
```

## Consequences

### ポジティブな影響

1. **知識ベースの動的拡張**: 学会ガイドライン改訂・施設プロトコルを
   ドキュメントアップロードだけでCIEに組み込める。

2. **Human Authority の完全維持**: AI抽出はpending/止まり。
   institutional/ への書き込みは常に人間承認後のみ。

3. **ライセンスフリーダム**: AbstractDocumentParser により、
   PyMuPDF（AGPL）から他ライブラリへの移行が1クラスの差し替えで完結する。

4. **後方互換性の完全保証**: Soft Delete + FrozenKnowledgeSet により、
   過去のworkflow実行は knowledge の状態変化に関わらず完全に再現可能。

5. **医療文献特性への対応**: related_entries の親子関係追跡により、
   ガイドライン改訂のチェーンを管理できる。

6. **最小コスト運用**: expires_at のアラートは UIロード時の差分計算のみ。
   バッチジョブ・スケジューラは不要。

### トレードオフ

1. **ディレクトリ複雑度の増加**: 3ネームスペース + pending/ の管理は
   ADR-0002と同様の複雑さをKnowledgeレイヤーにもたらす。

2. **knowledge/official/ への移行作業**: 現行の `knowledge/` 配下の
   ファイル群を `knowledge/official/` に移管する一時的な破壊的変更が発生する。
   この移行は実装PhaseのKIP-1で対処する。

3. **AI抽出精度の限界**: meta/knowledge-extractor/ の抽出精度は
   ドキュメントの品質・構造に依存する。
   confidence スコアと limitations の表示でユーザーに明示する。

---

## Forbidden（本ADRにより追加）

- `knowledge/official/` への直接書き込み（KnowledgeLifecycleServiceも禁止）
- `knowledge/institutional/` への KnowledgeLifecycleService 以外からの書き込み
- `approved_by_human: false` での知識エントリ作成
- 原典ファイルの物理削除
- PyMuPDF / 他パーサーライブラリへの KnowledgeIngestionAgent からの直接依存
- Planner Agent から SystemWorkflowRegistry の参照
- 実行中の FrozenKnowledgeSet の reload()
