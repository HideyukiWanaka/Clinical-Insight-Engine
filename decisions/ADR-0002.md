## Context

### 問題の所在

CIE PlatformのSkillは現時点で実質的にimmutableである。

1. **Skill更新フローが未定義**
   evaluation/によって生成されたスコア・発見事項がSkillにフィードバックされる
   仕組みが存在しない。Reviewerが毎回同じ問題を検出しても、Skillは改善されない。

2. **ユーザー固有のSkill拡張手段が未定義**
   施設固有の統計手法・報告フォーマット・図表スタイルをCIEに組み込む方法がない。
   全ユーザーが同一のSkillセットに縛られる。

3. **PROJECT_RULES.mdとの潜在的矛盾**
   Section 11「Skills never contain project-specific business logic」と
   「ユーザー固有Skillの追加」は一見矛盾する。

### 設計上の判断が必要な5点

| 問い | 本ADRでの決定 |
|-----|------------|
| Skill更新の主体 | AI提案 + 必ず人間承認（Human Authority必須） |
| 更新トリガー | evaluation結果の自動分析 + 手動リクエスト |
| ユーザーSkillの範囲 | `skills/user/`名前空間に隔離。コア名前空間は変更不可 |
| バージョニング | Skillごとにversion履歴を保持。旧バージョンは archived |
| PROJECT_RULES.md整合 | ユーザーSkillは「プロジェクト固有手順」であり業務ロジックではない。コアSkillは不変 |

## Decision

### 採用する設計: 3層Skill名前空間 + SkillLifecycleワークフロー

#### 原則1 — Skill名前空間の3分割

```
skills/
├── core/        ← CIE公式Skill（本ADR以前の全Skill）
│                   immutable: Anthropic/CIEチームのみ更新可
│                   例: statistics/t-test/, visualization/survival/
│
├── meta/        ← Skill改善・評価・更新を扱うMeta-Skill
│                   CIEが提供する自己改善インフラ
│                   例: meta/skill-evaluator/, meta/skill-proposer/
│
└── user/        ← ユーザー定義Skill
                    プロジェクト固有の手順・フォーマット・スタイル
                    Human Authority必須で追加・更新
                    例: user/my-hospital-table-format/
```

#### 原則2 — CoreSkillの不変性維持

`skills/core/`配下はユーザーが直接変更できない。
更新は必ずSkillLifecycle（ADR-0002で定義）を経由し、
CIEの開発プロセスとして管理される。
PROJECT_RULES.md Section 11の「Skills never contain project-specific
business logic」はcore Skillに適用される。

#### 原則3 — ユーザーSkillはcore Skillの拡張・オーバーライドが可能

ユーザーSkillはcoreと同一のinterface（SKILL.md構造）を持つ。
同一skill_idの場合、user/ が core/ より優先される（オーバーライド）。
ただし型の互換性（input/output schema）は必ず保持されなければならない。

#### 原則4 — AI提案はプロポーザルに留まる。実行は人間が承認

AIが評価結果からSkill改善案を自動生成することは許可されるが、
Skillファイルの実際の更新（書き込み）は必ず人間の明示的承認を経る。
PROJECT_RULES.md Section 3.7 Human Oversightの拡張実装。

#### 原則5 — Skillの進化は監査可能

すべてのSkill更新はimmutable audit logに記録される。
バージョン履歴は保持され、任意のバージョンにロールバック可能。

## Skill Lifecycle フロー

```
[Phase 1: 評価・検出]
  ─────────────────────────────────────────────────────
  workflow実行 → evaluation/生成 → regression.yaml検査
       │
       └─ skill_performance_score < threshold OR
          recurring_finding detected in reviewer output
               │
               ▼
  meta/skill-evaluator/ SKILL が起動
  → 対象Skillの問題を構造的に分析
  → SkillImprovementProposal を生成

[Phase 2: AI提案生成]
  ─────────────────────────────────────────────────────
  meta/skill-proposer/ SKILL が起動
  → 改善案のドラフト（SKILL.md diff形式）を生成
  → テストケース案を追加
  → SkillUpdateProposal スキーマに格納

[Phase 3: Human Review & Approval]
  ─────────────────────────────────────────────────────
  UIに SkillUpdateProposal を提示
  → [改善案を見る] → Diff viewer でSkill変更箇所を確認
  → [テストを実行] → 提案Skillのtests/を現データで検証
  → Human が 承認 / 修正 / 却下
       │
       ├─ 却下 → ProposalをARCHIVEDに遷移、理由を記録
       ├─ 修正 → HumanがSkill内容を直接編集後に承認
       └─ 承認 → Phase 4へ

[Phase 4: Skill更新・バージョニング]
  ─────────────────────────────────────────────────────
  旧バージョンを skills/core/{skill_id}/versions/{version}/ にアーカイブ
  新バージョンを skills/core/{skill_id}/SKILL.md に書き込み
  version を increment (e.g. 1.0.0 → 1.1.0)
  audit_log に SkillUpdateEvent を記録
  regression.yaml の baseline を更新（別途ADR-0001ルールに準拠）

[Phase 5: 継続モニタリング]
  ─────────────────────────────────────────────────────
  次回のworkflow実行から新Skillバージョンが使用される
  regression.yaml が新旧バージョンの性能を比較
  劣化した場合はロールバックをPropose
```

## User Skill追加フロー

```
[Step 1: テンプレート生成]
  ユーザーが「新しいSkillを追加」をリクエスト
  → meta/skill-scaffolder/ が SKILL.mdテンプレートを生成
  → skills/user/{user_skill_id}/ に配置

[Step 2: ユーザー記述]
  ユーザーがSKILL.md・examples/・tests/を記述
  （meta/skill-scaffolder/が提供するガイドに従う）

[Step 3: 検証]
  meta/skill-validator/ がSkill定義を検証:
  - SKILL.mdの必須セクションが存在するか
  - input/output schemaが既存スキーマと互換か
  - tests/に最低1件のテストケースがあるか
  - 業務ロジックが含まれていないか（PROJECT_RULES.md Section 11）

[Step 4: Human承認]
  検証結果をUIに提示 → Human承認
  → 承認後: skills/user/に正式登録
  → audit_log に UserSkillRegistrationEvent を記録

[Step 5: 有効化]
  次回のworkflow実行からuser Skillが利用可能になる
  core Skillとの競合がある場合: user Skillが優先（オーバーライド）
```

## Consequences

### ポジティブな影響

1. **評価→改善のフィードバックループ確立**
   evaluation/ の結果がSkillに還元される経路が生まれる。

2. **施設固有の知識をCIEに組み込み可能**
   施設のSOP・報告フォーマット・統計方針をuser Skillとして実装できる。

3. **Human Authority維持**
   AI提案はプロポーザル止まり。全Skill変更に人間承認が必須。

4. **CoreSkillの安定性維持**
   user/ 名前空間の分離により、コアSkillの品質が汚染されない。

### トレードオフ

1. **管理の複雑化**
   3つの名前空間の優先順位管理・バージョン履歴の保持はシステム複雑度を上げる。

2. **User Skillの品質保証困難**
   user Skillの内容はユーザー責任であり、CIEは形式的な検証のみ行う。
   統計的な妥当性はユーザーが確保する必要がある。
