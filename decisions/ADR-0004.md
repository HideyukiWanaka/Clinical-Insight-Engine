## Context

### 問題の所在

CIE Platformでは、PlannerAgentが毎回LLM APIを呼び出して
自然言語プロンプト → IntentObject への変換を行う。

この変換は以下の特性を持つ。

**特性1 — 同一入力に対して出力が安定している**

「A群とB群の比較をしたいです」という入力が与えられたとき、
データセットの列構造が同一であれば、PlannerAgentは
ほぼ確実に同じIntentObjectを返す。
これは統計的・臨床的概念への写像であり、創造的な生成ではない。

**特性2 — 繰り返し使用パターンが明確に存在する**

研究者は同一の研究テーマを複数のデータセットで繰り返し試す。
「A群とB群の比較」「治療前後の変化」のような典型的な表現は
使用するたびに同じLLM呼び出しを生成している。

**特性3 — LLM呼び出しがコストと速度のボトルネック**

無料枠（Gemini: 15 RPM）では1回の分析で
レート制限に達する場合がある。
有料プランでもAPIコールは積み上がるため、
気軽な試行錯誤が阻害される。

---

## Decision

### 採用する設計: 段階的セマンティックキャッシュ

使用するほどコストが下がる「育つキャッシュ」を
3フェーズで段階的に実装する。

---

### Phase 1: 完全一致キャッシュ（最初に実装）

**キャッシュキーの設計:**

```python
@dataclass(frozen=True)
class CacheKey:
    """
    キャッシュのキーはユーザー入力とデータ構造の両方を含む。
    データセットが異なれば変数マッピングが変わるため、
    入力文だけをキーにしてはならない。
    """
    normalized_prompt: str        # 小文字化・空白正規化・句読点除去
    dataset_fingerprint: str      # 列名エイリアス(var_N)と型の組み合わせのSHA-256
```

**正規化ルール（Phase 1）:**

```
「A群とB群の比較をしたいです」
「a群とb群の比較をしたいです」
「A群とB群の比較したい」
→ すべて同じキー: "a群とb群の比較をしたい"
```

具体的な正規化処理:
1. 全角→半角変換
2. ひらがな・カタカナは変換しない（意味が変わる場合があるため）
3. 小文字化（英数字のみ）
4. 末尾の助詞「を・が・は・です・ます」を除去
5. 連続空白を1つに圧縮

**キャッシュストレージ:**

```
cie/cache/
├── planner_cache.json     ← キャッシュ本体（キー→エントリのマップ）
└── cache_stats.json       ← ヒット数・コスト削減量の統計（UIに表示）
```

```json
// planner_cache.json の構造
{
  "version": "1.0",
  "entries": {
    "<sha256_of_cache_key>": {
      "cache_key": {
        "normalized_prompt": "a群とb群の比較をしたい",
        "dataset_fingerprint": "abc123..."
      },
      "original_prompts": [
        "A群とB群の比較をしたいです",
        "a群とb群の比較したいです"
      ],
      "intent_object": { ... },
      "confidence_score": 0.95,
      "created_at": "2026-07-01T10:00:00Z",
      "last_used_at": "2026-07-01T12:00:00Z",
      "use_count": 3,
      "llm_provider": "anthropic",
      "llm_model": "claude-haiku-4-5-20251001"
    }
  }
}
```

### Phase 2: 正規化拡張（Phase 1の後）

Phase 1の正規化に加えて、同義語・表記ゆれを吸収する辞書を導入する。

```python
# cie/cache/normalization.py
SYNONYM_MAP: dict[str, str] = {
    # 比較系
    "比べたい": "比較したい",
    "差を見たい": "比較したい",
    "違いを調べたい": "比較したい",
    "compare": "比較",
    # 前後系
    "ビフォーアフター": "前後比較",
    "before after": "前後比較",
    "介入前後": "前後比較",
    "治療前後": "前後比較",
    # 相関系
    "関係を見たい": "相関を調べたい",
    "関連を見たい": "相関を調べたい",
    # （使用実績から随時追加）
}
```

この辞書は `original_prompts` の蓄積から
人間がキュレーションして追加していく。

---

### Phase 3: ベクトル類似度キャッシュ（Phase 2の後）

意味が近いプロンプトをベクトル類似度で検出する。

```python
# Phase 3 は埋め込みモデルが必要
# 実装時点で軽量な選択肢を選ぶ（例: sentence-transformers のローカルモデル）
# オンライン埋め込みAPIは使わない（オフラインファーストの原則 S.3.4）

SIMILARITY_THRESHOLD = 0.92  # この値以上なら同一意図とみなす
```

Phase 3はオフラインファースト原則（PROJECT_RULES.md S.3.4）に従い、
ローカル埋め込みモデルのみ使用する。
外部埋め込みAPIへの依存は禁止する。

## PlannerAgent への組み込み方針

```python
# cie/agents/planner.py の _execute() に追加する処理フロー

async def _execute(self, agent_input: AgentInput) -> AgentOutput:
    # ... （既存の PII チェック）

    # [新規] Step 1.5 — キャッシュ参照
    cache_key = self._cache_store.make_key(
        prompt=user_prompt,
        dataset_metadata=dataset_metadata,
    )
    cached = self._cache_store.get(cache_key)
    if cached is not None:
        # キャッシュヒット: LLM を呼ばずに返す
        self._cache_store.record_hit(cache_key)
        return self._build_output_from_cache(agent_input, cached)

    # ... （既存の LLM 呼び出し）

    # [新規] Step 3.5 — キャッシュ書き込み（LLM成功時のみ）
    self._cache_store.put(
        key=cache_key,
        original_prompt=user_prompt,
        intent_object=intent_obj,
        confidence_score=confidence_score,
        llm_provider=self._llm_client.provider,
        llm_model=self._llm_client.model,
    )
```

## キャッシュのガバナンスルール

| ルール | 内容 |
|--------|------|
| **CA-001** | キャッシュヒット時もAuditLogに `CACHE_HIT` イベントを記録する |
| **CA-002** | confidence_score が 0.7 未満のエントリはキャッシュしない |
| **CA-003** | requires_human_clarification=true の結果はキャッシュしない（曖昧な入力は再分類が必要） |
| **CA-004** | キャッシュエントリの手動削除はUIから可能。物理削除を許可する（FrozenKnowledgeSetとは異なる） |
| **CA-005** | キャッシュは LLM モデルをまたいで共有しない（モデル変更時は別エントリ） |
| **CA-006** | dataset_fingerprintが変わればキャッシュミス扱いとする |
